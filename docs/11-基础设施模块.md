# 基础设施模块

## 1. 模块目标

- 为智能小说平台提供可观测性与可运维能力：日志、指标、健康检查、异常治理、定时任务与告警闭环。
- 统一“配置、错误码、响应格式、审计事件、TraceId”规范，降低跨模块协作与排障成本。
- 在单体阶段以最小可用集落地，保证后续横向扩展（拆分服务、引入 MQ/对象存储/外部审计）时不重写核心约束。

## 2. 职责边界

模块负责：

- 日志与追踪：结构化日志字段规范、TraceId 贯通、审计日志要求。
- 监控与健康：Actuator 指标暴露、健康检查、关键 SLA 指标与告警条件。
- 任务调度：定时任务约束、幂等与并发控制、失败重试与补偿。
- 异常治理：统一异常响应、错误码分层、异常分类与降级约束。
- 配置治理：分环境配置与敏感配置管理约束。

模块不负责：

- 各业务模块的具体业务逻辑与数据一致性实现细节（各模块文档自行定义）。
- 安全认证授权逻辑实现（由安全治理模块负责）；本模块仅补充审计与配置安全约束。
- 缓存具体策略实现（由缓存模块负责）；本模块仅定义监控指标与运行约束。

## 3. 模块结构

建议包结构（目标态，后续实现时落到代码包结构）：

- `com.ainovel.infrastructure`
  - `log`：日志规范、TraceId、审计事件定义
  - `monitor`：Actuator 配置、指标与健康检查约束
  - `task`：定时任务调度与通用幂等/锁约束
  - `exception`：统一异常、错误码与响应封装
  - `config`：配置分层与环境隔离约束

说明：当前阶段只产出文档与约束，不落代码。

## 4. 核心方案

## 4.1 日志与追踪方案

### 4.1.1 结构化日志字段（必须）

每条请求日志至少包含：

- `ts`：时间戳
- `level`：日志级别
- `traceId`：链路标识（每个请求必有）
- `requestId`：请求唯一标识（可与 traceId 相同或衍生）
- `method`、`path`、`status`、`elapsedMs`
- `userId`（可选，已登录才有）、`role`（可选）
- `clientIp`、`userAgent`（可选）

业务审计日志（敏感操作）额外必须包含：

- `action`：如 `NOVEL_PUBLISH`、`NOVEL_BAN`、`AUDIT_MANUAL_DECIDED`
- `bizType`、`bizId`
- `fromStatus`、`toStatus`（若有状态流转）
- `reason`（必要时，禁止包含敏感快照全文）

### 4.1.2 日志等级与内容约束

- `INFO`：请求摘要、核心业务事件（不包含敏感内容）。
- `WARN`：可恢复异常、风控/限流触发、状态机不匹配、重试等。
- `ERROR`：不可恢复异常、依赖不可用、数据损坏风险（必须带 traceId 与关键上下文）。

禁止写入日志的内容：

- 明文密码、JWT、Refresh Token、验证码。
- 审核 `content_snapshot` 原文、敏感词命中细节、用户隐私字段（身份证/手机号全量等）。

### 4.1.3 TraceId 贯通

- 接入层或应用层生成 `traceId`，通过 HTTP header 贯通（如 `X-Trace-Id`）。
- 异步任务必须传递并生成可关联的 `parentTraceId`（或在日志中记录 `triggerTraceId`）。

## 4.2 监控与健康检查方案

### 4.2.1 Actuator 暴露范围（建议）

- 健康检查：`/actuator/health`
- 指标：`/actuator/metrics`
- 应用信息（可选）：`/actuator/info`

约束：

- 生产环境必须加访问控制（内网、白名单、鉴权）。
- 不暴露敏感配置与堆栈到公网。

### 4.2.2 关键指标（必须可观测）

- HTTP：`p50/p95/p99` 延迟、`5xx` 比例、Top N 慢接口
- JVM：堆内存、GC 次数与耗时、线程数
- 数据库：连接池活跃数、超时/失败次数、慢 SQL 计数（至少能定位）
- Redis：命中率（L2）、失败率、耗时分布
- 缓存防护：穿透拦截次数、互斥锁竞争次数（若实现）
- 任务队列：积压长度、处理耗时、失败次数（审核/推荐/消息等）

### 4.2.3 健康检查判定

- `UP`：依赖可用且关键指标不触发熔断阈值
- `DEGRADED`（建议）：依赖可用但性能/失败率异常（用于告警）
- `DOWN`：数据库不可用或关键依赖不可用

## 4.3 定时任务与异步任务方案

### 4.3.1 任务类型（目标态）

- 推荐刷新、榜单重算
- 审核积压扫描与 SLA 超时检查
- 失效缓存预热（热点书籍、热门章节）
- 统计聚合与归档（评论、消息、审核快照等）

### 4.3.2 任务约束（必须）

- 幂等：同一批次任务重复执行不会导致重复写入或重复状态流转。
- 并发控制：同一任务在同一时间只允许一个实例执行（单体阶段可用 JVM 锁；后续演进为 Redis 锁）。
- 超时与重试：每个任务定义最大执行时间与最大重试次数，重试必须指数退避或固定间隔。
- 失败可见：任务失败必须写结构化日志并可告警，必要时落“任务失败表”（后续实现）。

### 4.3.3 与业务模块协作边界

- 任务只触发“刷新/重算/补偿”，不得绕开业务模块状态机直接写核心业务表。
- 对“发布/下架/复审”等敏感状态变更的补偿，必须通过业务模块公开的用例入口或任务表驱动。

## 4.4 异常治理方案

### 4.4.1 统一响应与错误码分层

统一响应：

```json
{ "code": 0, "message": "success", "data": {} }
```

错误码分层（全局一致）：

- `1000-1999`：参数错误
- `2000-2999`：认证授权错误
- `3000-3999`：业务状态错误（状态机不允许、幂等冲突等）
- `5000-5999`：系统内部错误（依赖失败、超时、未知异常）

### 4.4.2 异常分类与处理约束

- 参数校验失败：直接返回 1xxx，不记录 ERROR（最多 WARN）。
- 权限不足：返回 2xxx，必须记录审计（至少记录 action/path/userId）。
- 状态机不允许：返回 3xxx，必须记录 from/to 状态与 bizId。
- 依赖失败（DB/Redis）：返回 5xxx，必须告警并包含 traceId。

降级约束（设计层要求）：

- 推荐接口可降级到“热榜/分类规则”。
- 审核受理不可用时，上游应降级为“进入待审队列”或“阻断发布”（由业务模块决定并在其文档声明）。

## 5. 接口与配置约束

### 5.1 配置分层

- `application.properties` 仅保留最小默认配置。
- 增加分环境配置约束（目标态）：`application-dev.properties`、`application-prod.properties`（或 YAML 分 profile）。

### 5.2 必要配置项（目标态约束）

- MySQL：URL、用户名、密码（禁止硬编码到仓库）
- Redis：地址、密码（如有）、连接池参数
- Token：密钥、过期时间、刷新策略
- 日志：输出格式（JSON/Key-Value）、日志级别、采样策略（若需要）
- 任务：线程池大小、队列长度、重试次数与间隔

### 5.3 配置安全

- 禁止提交真实密钥与生产连接串。
- 敏感配置使用环境变量或外部注入（如容器环境、CI Secrets）。

## 6. 运行与告警约束

必须告警的场景（建议阈值后续按压测调整）：

- HTTP `5xx` 比例异常（按接口维度 + 全局）
- `p99` 延迟持续超阈值
- DB 连接池耗尽、超时率异常
- Redis 失败率异常
- 审核/推荐/消息任务积压超阈值
- 任务失败连续 N 次或超过最大重试

运行约束：

- 所有告警必须能定位到：`traceId`、`service`、`module`、`endpoint/taskName`、`elapsedMs`、`errorCode`。
- 发生系统性故障时，必须优先保证“读可用”，写操作可降级或保护性失败（由各模块声明）。

## 7. 安全与审计约束

- 对以下操作必须记录审计事件与操作者信息：
  - 登录失败、权限拒绝
  - 小说发布/下架/封禁
  - 审核人工复审结论
  - 角色与权限变更（如后续加入）
- 审计日志必须可检索，且不包含敏感内容快照全文。

## 8. 实现边界

- 本模块提供“规范、约束、统一能力”的设计，不实现业务功能。
- 业务模块必须依赖本模块的错误码、响应格式与日志字段规范，但不得把业务逻辑塞入基础设施层。
- 与安全治理、缓存、持久化模块之间只通过约束对齐，不形成循环依赖。

## 9. 开发顺序

1. 统一 `Result<T>` 响应与全局错误码分层，定义基础异常类型与映射规则。
2. 定义 TraceId/RequestId 贯通规范与结构化日志字段，确定禁止记录的敏感字段清单。
3. 接入 Actuator 并明确生产环境访问控制策略，梳理关键指标与告警条件。
4. 定义定时任务规范：幂等、并发控制、超时、重试与失败可见性。
5. 建立审计事件清单与最小字段集合，并在关键模块文档中对齐 action 命名。
6. 明确配置分环境与密钥管理规范，为后续上线做准备。
