# 推荐引擎模块

## 1. 模块目标

- 提供稳定、可扩展的推荐能力：`首页推荐`、`相关推荐`、`继续阅读/追更`、`分类推荐（预留）`。
- 将推荐计算从在线请求链路剥离，采用“异步计算 + 缓存返回”为主，确保推荐接口低延迟、可用性优先。
- 提供冷启动与降级策略，在画像缺失、缓存未命中或下游异常时仍可返回可用结果。

## 2. 职责边界

模块负责：

- 生成并缓存推荐结果（以小说 ID 列表为主），对外提供查询接口。
- 维护轻量用户画像/偏好（规则版）与推荐结果元数据（场景、版本、生成时间、算法标识）。
- 异步刷新：定时刷新与事件触发刷新（只做失效/排队，不同步重算）。
- 推荐相关日志与指标规范：命中率、过滤率、计算耗时、失败率（只定义约束）。

模块不负责：

- 小说/章节主数据的创建、编辑与状态流转（小说管理模块）。
- 阅读进度的权威记录与章节正文读取优化（阅读模块）。
- 评论/点赞/收藏/关注等互动行为的权威记录（社区互动模块）。
- 审核规则与人工复审（审核引擎模块）；推荐仅消费“可见性/可推荐”结论。

禁止依赖：

- 禁止直接操作其它业务域内部表；推荐只能通过“公开查询接口”或“汇总表/视图”获取必要特征（后续实现落地）。
- 禁止在在线请求中触发大范围扫描数据库或同步重计算。

## 3. 模块结构

建议包结构（以 `com.ainovel.recommend` 为根）：

- `controller`：推荐查询接口（用户端）。
- `service`
  - `RecommendQueryService`：在线查询聚合、缓存读取、二次过滤、降级。
  - `RecommendComputeService`：规则推荐计算、结果写入缓存/存储。
- `domain`
  - `RecommendScene`：场景枚举（`HOME`, `RELATED`, `CONTINUE`, `CATEGORY`）。
  - `RecommendResult`：推荐结果（item 列表 + 版本 + 生成/过期信息）。
  - `UserProfileLite`：轻量画像（偏好分类/标签、近期行为摘要）。
- `repository`：画像与结果存储访问层（初期 Redis 为主，MySQL 预留）。
- `task`：定时任务与异步刷新入口（Spring Task / `@Async`）。
- `converter`：领域对象与存储对象转换。

## 4. 核心流程

### 4.1 在线查询（读链路）

以首页推荐为例：

1. 识别用户态（登录/匿名），解析 `scene` 与 `size`。
2. 构造缓存 Key（见 7.1），读取 Redis 推荐结果。
3. 命中后做轻量过滤：
   - 过滤不可见/下架/封禁内容（依赖小说管理模块的可见性查询/摘要缓存）。
   - 去重、截断到 `size`，不足时走补齐策略（见 8.1）。
4. 缓存未命中或结果过期：返回降级结果，并触发异步补算（不阻塞请求）。

约束：

- 在线请求不得触发同步计算；允许投递异步任务。
- 必须限制 `size`：默认 20，上限 50。
- 只返回摘要卡片信息，不返回大字段（章节正文、审核细节等）。

### 4.2 异步计算（规则版起步）

计算分两步：召回（Candidate）与排序（Rank）。

召回来源（初期）：

- 热度召回：近 24h/7d 阅读量、收藏量、评论量加权（权重配置化）。
- 分类/标签召回：基于用户画像偏好与小说分类/标签匹配。
- 新书与更新：最近更新时间、更新频率加权。
- 继续阅读：读取阅读模块提供的“最近阅读小说”摘要进行召回。
- 相关推荐：基于当前小说的分类/标签召回同类内容。

排序规则（初期）：

- 可见性过滤：必须 `ON_SHELF` 且未封禁，审核可见。
- 规则打分：`score = w_hot*hot + w_match*tagMatch + w_fresh*freshness + w_div*diversityPenalty`。
- 多样性约束：同作者/同标签/同分类占比上限（建议不超过 40%）。

输出：

- 写入 Redis：只写小说 ID 有序列表与元数据（生成时间/过期时间/版本/算法标识）。

### 4.3 刷新触发

- 定时刷新：
  - 匿名首页推荐与热榜：每 10-30 分钟（可配置）。
- 事件触发（只失效/排队）：
  - 小说上架/下架/封禁：失效相关场景缓存或标记逻辑过期。
  - 热度特征变化显著：排队触发热榜重算（异步）。

约束：

- 事件触发禁止同步重算；只允许投递任务或设置“逻辑过期”。

## 5. 接口约束

通用约束：

- 统一前缀：`/api/v1/**`。
- 统一响应：`Result<T>`。
- 读接口天然幂等；任何“刷新/重算”管理端接口必须幂等。
- 限流：按 `IP + userId + path`，匿名更严格（由安全治理模块提供能力）。

建议接口：

- 首页推荐
  - 路径：`GET /api/v1/recommend/home?size=20`
  - 用途：获取首页推荐列表
  - 权限：可匿名（匿名走热门规则）
  - 关键约束：`size` 上限 50；必须做可见性过滤；命中缓存不得触发同步计算

- 相关推荐
  - 路径：`GET /api/v1/recommend/novels/{novelId}/related?size=20`
  - 用途：获取同类/相似小说推荐
  - 权限：可匿名
  - 关键约束：`novelId` 不可见时返回空列表或降级热门（配置化）

- 继续阅读/追更推荐（可选）
  - 路径：`GET /api/v1/recommend/continue?size=20`
  - 用途：基于阅读历史给出继续阅读/追更
  - 权限：必须登录
  - 关键约束：必须依赖阅读模块汇总信息，禁止直接扫章节表

返回数据约束：

- 推荐接口只返回“卡片摘要”：`novelId/title/coverUrl/authorName/category/tagsSummary/hotScore(optional)`。
- 禁止返回用户画像明细、互动明细、审核命中规则等敏感信息。

## 6. 数据约束

### 6.1 用户画像（轻量版）

目标：初期不引入复杂画像系统，使用轻量聚合支持规则推荐。

建议结构（逻辑实体）：`user_profile_lite`

- `user_id`：用户 ID
- `prefer_categories`：偏好分类 Top N
- `prefer_tags`：偏好标签 Top N
- `recent_novels`：近期阅读/收藏小说 Top N
- `last_updated_at`：更新时间
- `version`：画像版本

约束：

- 画像更新为异步最终一致，允许延迟（建议 10-60 分钟）。
- 画像来源只读：阅读/社区模块提供汇总数据（接口或汇总表形式，后续实现）。

### 6.2 推荐结果

建议结构（逻辑实体）：`recommend_result`

- `scene`：场景
- `subject`：主体（`userId` 或 `novelId` 或 `anon`）
- `item_ids`：推荐小说 ID 列表（有序）
- `generated_at`：生成时间
- `expire_at`：逻辑过期时间
- `version`：推荐版本
- `algorithm`：算法标识（初期 `rule_v1`）

约束：

- 必须可追溯：至少保留 `generated_at/version/algorithm`。
- `item_ids` 中禁止出现不可见内容；若后续变为不可见，在线查询必须二次过滤并触发异步补算。

## 7. 缓存、事务与一致性约束

### 7.1 Redis Key 规范

统一约束：

- Key 必须包含环境前缀（由配置注入），防止多环境串用。
- TTL 必须带随机抖动，避免雪崩：`ttl = baseTtl + rand(0..jitter)`。

Key（建议）：

- 首页推荐：
  - 匿名：`recommend:home:anon:v{ver}`
  - 登录：`recommend:home:{userId}:v{ver}`
- 相关推荐：`recommend:related:{novelId}:v{ver}`
- 继续阅读：`recommend:continue:{userId}:v{ver}`
- 画像：`recommend:profile:{userId}:v{ver}`

### 7.2 写入与失效策略

模式：Cache Aside（推荐计算写入缓存；在线查询只读缓存）。

- 写入：一次写入完整列表与元数据（版本/生成时间/过期信息/算法标识）。
- 失效：
  - 强失效：删除 Key（适用于封禁、下架等强一致可见性变化）
  - 逻辑过期：保留旧值但标记过期（在线可返回旧值并触发异步刷新）

一致性约束：

- 推荐与可见性之间采用最终一致：在线查询必须过滤不可见项；过滤导致数量不足允许短返回或用热门补齐。

### 7.3 计算任务去重

- 同一 `scene + subject + version` 的计算任务必须去重，避免并发重复计算。
- 去重实现建议：Redis 分布式锁或任务状态表（具体实现由缓存/基础设施模块落地）。

## 8. 降级策略

### 8.1 在线降级（强制可用）

降级优先级（从高到低）：

1. 返回未过期缓存结果。
2. 返回逻辑过期缓存结果（同时触发异步刷新）。
3. 返回匿名热门列表（热度规则缓存）。
4. 返回空列表（仅在可见性/数据严重异常时允许）。

补齐策略：

- 若过滤后数量不足，优先用“同分类热门”或“全站热门”补齐。
- 补齐内容同样必须经过可见性过滤与去重。

### 8.2 计算降级

- 画像缺失：按热门 + 分类召回（匿名策略）计算。
- 下游汇总数据不可用：仅使用热度榜与新书更新规则。
- 计算失败：保留旧缓存（逻辑过期），并按退避策略重试。

## 9. 异常、日志与审计约束

日志字段规范：

- 在线查询（采样）：`traceId`, `scene`, `subject`, `hit`(hit/miss), `size`, `filteredCount`, `latencyMs`, `degradeLevel`
- 计算任务：`taskId`, `scene`, `subject`, `algorithm`, `candidateSize`, `resultSize`, `durationMs`, `status`
- 异常：必须包含 `scene/subject/version/algorithm`，避免无法定位

错误处理：

- 推荐查询异常不得影响主链路：必须降级返回，并记录告警。
- 计算任务失败必须可重试，并有最大重试次数与退避策略（由基础设施模块统一约束）。

告警建议：

- miss 率异常升高、过滤比例异常升高、计算失败率升高、计算耗时突增。

## 10. 实现边界

- 初期仅实现规则推荐（`rule_v1`），不引入向量检索、在线学习或复杂召回/排序系统（后续迭代）。
- 初期允许“Redis 为主，MySQL 备份可选”的存储策略，但元数据必须可追溯。
- 不在本模块内实现全站埋点系统，仅定义事件与字段规范，后续由基础设施统一落地。

## 11. 开发顺序

1. 定义推荐场景枚举、推荐结果 VO、Key 规范与配置项（TTL、size 上限、版本）。
2. 实现在线查询服务：Redis 读取 + 可见性过滤 + 降级返回 + 异步补算触发（不阻塞）。
3. 实现规则版计算服务：热度召回 + 标签/分类召回 + 规则排序 + 多样性约束。
4. 接入异步刷新：定时刷新匿名首页/热榜；事件触发仅做失效/排队。
5. 补齐轻量画像更新：从阅读/社区汇总数据拉取，异步更新画像缓存。
6. 增加监控与告警：命中率、延迟、过滤比例、计算耗时、失败率。
