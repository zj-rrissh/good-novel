# 安全治理模块

## 1. 模块目标

为平台提供统一的认证授权、请求治理与安全防护能力，保证所有对外 API 的访问可控、可审计、可追溯，并对常见攻击与滥用行为（爆破、刷接口、爬取、注入、XSS）进行拦截或降级。

## 2. 职责边界

模块负责：

- 认证：登录凭证校验、Access Token/JWT 校验、登出与 Token 失效。
- 授权：RBAC 权限模型、接口鉴权、管理端与用户端隔离。
- 请求治理：全局过滤器、TraceId 注入、基础安全头、输入清洗。
- 限流与风控：按 IP/用户/设备/接口维度限流、异常行为记录与封禁策略。
- 审计：高风险操作日志、鉴权失败原因与限流触发原因可追溯。

模块不负责：

- 业务层具体权限规则的“内容含义”（如“作者是否能编辑某小说”属于业务校验，安全模块只提供鉴权框架与通用策略）。
- Nginx 层的限流与 WAF（可协作，但不在本模块内实现）。
- 内容审核（由审核引擎模块负责）。

依赖与协作：

- 依赖用户中心模块提供用户账号状态（禁用/冻结/角色集合）与登录接口。
- 依赖 Redis（可选）保存 Token 黑名单、限流计数与风控状态。
- 与基础设施模块共享统一异常与统一响应 `Result<T>`、日志规范与 TraceId 规范。

## 3. 模块结构

建议包结构（示例）：

```text
com.ainovel.security
├── config                # Security 配置、白名单、CORS
├── auth
│   ├── token             # JWT 生成/解析/校验、Key 管理
│   ├── rbac              # 角色/权限映射与鉴权决策
│   └── context           # 当前用户上下文（ThreadLocal/Request scope）
├── filter                # 过滤器链：TraceId/日志/XSS/鉴权/限流
├── rate
│   ├── limiter           # 限流器实现（Redis/本地）
│   └── policy            # 限流策略与配置解析
├── risk                  # 风控规则、封禁与异常行为记录
└── audit                 # 安全审计事件与记录输出
```

核心配置来源：

- `application.properties`/分环境配置：JWT 过期时间、密钥、白名单路径、限流阈值、风控开关。
- 数据源（可选）：RBAC 权限字典与角色映射配置（后续可运营化）。

## 4. 核心流程

### 4.1 登录认证流程（用户中心调用链）

1. 用户中心完成用户名/密码校验与账号状态校验（禁用/冻结）。
2. 安全模块签发 Access Token（JWT），并返回给客户端。
3. 若启用刷新机制：签发 Refresh Token，存入 Redis（与设备绑定）并设置过期。

约束：

- Access Token 必须短期有效（建议 1-2 小时），并包含 `userId`、`roles`、`exp`、`jti`、`tokenVersion`。
- 账号冻结/角色变更后必须可使旧 Token 失效（通过 `tokenVersion` 或黑名单实现）。

### 4.2 请求鉴权流程（通用 API）

1. 进入全局过滤器链，生成/透传 `traceId`。
2. 对白名单路径直接放行（静态资源、登录、注册、健康检查等）。
3. 提取并校验 `Authorization: Bearer <token>`。
4. 校验通过后写入当前用户上下文（userId、角色、来源信息）。
5. 进入 RBAC 鉴权决策（路径/方法 -> 权限点）。
6. 进入业务 Controller；业务层仍需做资源级权限校验（如作者是否拥有该小说）。

### 4.3 限流与风控流程

1. 在鉴权前或鉴权后执行限流（建议：对匿名接口在鉴权前，对用户态接口在鉴权后）。
2. 维度选择：IP、用户 ID、设备 ID、接口路径、HTTP 方法。
3. 触发限流返回统一错误码与可观测信息（但不暴露内部策略细节）。
4. 连续触发阈值进入风控：写入 Redis 封禁 Key，并输出审计日志。

## 5. 接口约束

本模块对外不直接暴露业务 API，但对平台接口有统一约束：

- API 版本前缀：`/api/v1/**`。
- 统一响应：`Result<T>`；鉴权失败/限流必须返回统一错误码。
- Header 规范：
  - `Authorization: Bearer <JWT>`
  - `X-Request-Id`（可选，客户端传入；否则服务端生成）
  - `X-Device-Id`（建议，H5/APP 传入用于风控）
- 幂等性：
  - 登录接口：同账号同设备短期重复提交需防爆破（配合限流），不强制幂等。
  - 写接口（评论/点赞/发布章节等）：必须配合限流与业务幂等（由业务模块实现）。

白名单路径建议：

- `/api/v1/auth/login`
- `/api/v1/auth/register`
- `/actuator/health`（如启用）
- 静态资源路径由 Nginx 处理，不进入应用鉴权链路

管理端路径约束：

- 所有管理端接口统一前缀：`/api/v1/admin/**`。
- `ADMIN`（或等价角色）必须作为最小访问门槛；更细粒度权限点按功能拆分。

## 6. 数据约束

### 6.1 JWT Claims 约束

必需字段：

- `sub`：用户唯一标识（建议存 userId 字符串）。
- `roles`：角色列表（字符串数组）。
- `exp`：过期时间戳。
- `iat`：签发时间戳。
- `jti`：Token 唯一 ID（用于黑名单与审计关联）。
- `tv`：tokenVersion（用于强制失效）。

约束：

- 不在 JWT 中存放敏感信息（明文手机号、密码摘要等）。
- roles 只作为粗粒度授权依据；资源级权限必须业务侧二次校验。

### 6.2 RBAC 数据模型（逻辑约束）

- 角色：`USER`、`AUTHOR`、`ADMIN`（初期可固定枚举，后续可扩展为表驱动）。
- 权限点：按“模块:动作”命名，例如 `novel:write`、`audit:review`、`user:ban`。
- 映射规则：`(method, pathPattern) -> permission`，支持方法级别区分。

## 7. 缓存、事务与一致性约束

Redis 使用建议（可选但推荐）：

- Token 黑名单：
  - Key：`auth:blacklist:{jti}`
  - TTL：与 token 剩余有效期一致
  - 用途：登出/封禁时立刻失效
- Refresh Token（若启用）：
  - Key：`auth:refresh:{userId}:{deviceId}`
  - TTL：长于 Access Token（如 7-30 天）
- tokenVersion：
  - Key：`auth:tv:{userId}`
  - 用途：角色/状态变更时递增，强制旧 token 失效
- 限流计数：
  - Key：`rate:{dimension}:{id}:{path}:{window}`
  - TTL：窗口大小

一致性约束：

- 若同时启用黑名单与 tokenVersion：校验顺序必须固定（先 `tv`，再黑名单或反之）并在文档中保持一致，避免灰度期间出现放行/误杀不一致。
- 账号状态变更（禁用/冻结）必须触发 `tokenVersion` 递增或黑名单写入，确保已登录会话在可控时间内失效。

事务约束：

- 安全模块本身不持有业务事务；与用户中心联动的登录/登出流程如涉及数据库更新，事务由用户中心负责。

## 8. 安全约束

### 8.1 基础防护

- XSS：对富文本字段由业务层做白名单清洗；安全模块对通用参数做基础转义/校验，避免反射型 XSS。
- SQL 注入：禁止拼接 SQL；安全模块可做简单模式拦截（高风险字符/语句），但最终依赖规范化数据访问层（MyBatis 参数绑定）。
- CORS：默认关闭跨域，若需要仅允许指定 Origin；禁止 `*` + `credentials=true` 的组合。
- 安全头：建议统一添加 `X-Content-Type-Options`、`X-Frame-Options`、`Content-Security-Policy`（按前端需求配置）。

### 8.2 凭证与密钥

- JWT 签名密钥不得硬编码，必须从环境变量或分环境配置加载。
- 密钥轮换需有兼容窗口（双 Key 验签），并记录轮换审计。

### 8.3 账号与权限安全

- 登录失败必须进行延迟与限流（防爆破），并输出审计日志但不回显具体失败原因给客户端（避免用户名枚举）。
- 管理端接口必须二次校验权限点，不允许仅凭路径前缀放行。

## 9. 异常、日志与审计约束

### 9.1 统一错误码建议（与全局异常模块对齐）

- `2001`：未认证（缺少/无效 Token）
- `2002`：Token 过期
- `2003`：无权限（RBAC 拒绝）
- `2004`：账号被禁用/冻结
- `2101`：限流触发
- `2102`：风控封禁

### 9.2 日志规范

必须记录字段：

- `traceId`、`requestId`、`path`、`method`、`clientIp`、`userId`（若已认证）、`deviceId`（若有）
- 鉴权结果：`auth=pass|fail`，失败原因枚举（内部）
- 限流结果：`rate=pass|block`，命中策略 ID（内部）

审计事件（至少覆盖）：

- 登录成功/失败（含失败类型：密码错误/账号不存在/账号禁用/验证码失败等）
- 角色变更、封禁/解封
- 管理端高危操作（删除小说、审核驳回、用户禁用等）
- 触发风控封禁（含维度、阈值、封禁时长）

约束：

- 客户端返回信息保持克制，不泄露策略阈值、内部权限点或黑名单命中信息。
- 任何鉴权/限流异常都必须有日志可查，且与 `traceId` 关联。

## 10. 实现边界

- 安全模块只提供框架与通用能力，不实现具体业务授权（资源级权限）逻辑。
- 风控封禁只做“是否允许访问”的通用判断，不做业务处罚策略（如扣积分）等。
- 限流策略初期配置化即可，后续运营化（管理后台动态调整）属于增强项。

## 11. 开发顺序

1. 定义统一错误码与 `Result<T>` 约定（与基础设施模块对齐）。
2. 引入 Spring Security 基础配置与白名单机制，完成 JWT 解析/校验与上下文注入。
3. 建立 RBAC：角色枚举、权限点命名规则、路径到权限映射表。
4. 接入全局过滤器：TraceId、访问日志、基础输入清洗。
5. 接入限流器（优先 Redis 计数），完成匿名与用户态两套策略。
6. 接入风控封禁（Redis Key），实现封禁检查与审计输出。
7. 补齐管理端接口隔离策略与审计事件覆盖面。
