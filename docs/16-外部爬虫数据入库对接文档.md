# 外部爬虫数据入库对接文档

## 1. 文档目标

本文档面向“在仓库外部单独开发自动化爬虫程序”的场景，说明如何将抓取到的小说数据直接写入本项目数据库。

适用目标：

- 外部 Python / Java / Node.js 爬虫程序
- 数据采集后直接写 MySQL
- 不走本项目作者端“建书 / 建章 / 提审 / 上架”接口

本文档覆盖：

- 需要写入的数据库表
- 字段含义与约束
- 状态枚举约定
- 推荐入库顺序
- 去重与更新策略
- 示例 SQL

不覆盖：

- 站点反爬绕过
- 登录态抓取
- 分布式调度
- JS 动态渲染抓取

## 3. 外部爬虫建议只操作哪些表

外部爬虫程序推荐只直接写以下两张核心表：

- `novel`
- `novel_chapter`

外部爬虫通常只读、不写以下表：

- `user_account`
- `user_profile`
- `audit_task`
- `reading_progress`
- `comment`
- `reaction`
- `user_follow`
- `user_message`

原因：

- 小说展示和阅读主数据只依赖 `novel` 与 `novel_chapter`
- 其他表属于用户行为、审核、消息和运营数据，不应由爬虫伪造

## 4. 导入前置条件

### 4.1 作者账号

`novel.author_id` 不能为空，因此必须先确定一个“归属作者 ID”。

当前开发库默认有以下作者账号：

- `author_demo`
- `user_account.id = 1002`

建议：

- 开发环境直接使用 `author_id = 1002`
- 生产或正式环境创建专用“内容导入作者”账号，再把该账号的 `id` 固定给爬虫使用

### 4.2 字符集

数据库脚本使用：

- `utf8mb4`
- `utf8mb4_0900_ai_ci`

爬虫程序写库时也必须使用 UTF-8，避免中文标题、正文乱码。

### 4.3 时间格式

表内时间字段使用 `DATETIME(3)`，精度到毫秒。

建议外部程序写入格式：

- `YYYY-MM-DD HH:MM:SS.SSS`

例如：

- `2026-03-23 14:30:00.000`

## 5. 核心表结构说明

### 5.1 `novel` 主表

用途：

- 保存小说级别的摘要信息和聚合信息

建表关键约束：

- 主键：`id`
- 无“作者 + 标题”唯一索引
- 由业务侧自行控制去重

字段说明：

| 字段 | 类型 | 必填 | 说明 | 外部爬虫写入建议 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT` | 否 | 自增主键 | 不传，让数据库生成 |
| `author_id` | `BIGINT` | 是 | 作者用户 ID | 推荐固定为导入作者 ID，如 `1002` |
| `title` | `VARCHAR(200)` | 是 | 小说标题 | 去首尾空格，建议不超过 200 字符 |
| `intro` | `TEXT` | 否 | 小说简介 | 可空 |
| `cover_url` | `VARCHAR(512)` | 否 | 封面 URL | 可空，建议绝对 URL |
| `category_id` | `BIGINT` | 否 | 分类 ID | 当前库未强制外键，可空 |
| `tag_ids` | `VARCHAR(255)` | 否 | 标签 ID 列表，逗号分隔 | 推荐升序存储，如 `1,2,5`；无标签可写 `''` 或 `NULL` |
| `status` | `VARCHAR(32)` | 是 | 小说状态 | 外部导入推荐写 `ON_SHELF` |
| `latest_chapter_id` | `BIGINT` | 否 | 最新已发布章节 ID | 插入章节后回填 |
| `word_count` | `BIGINT` | 是 | 已发布章节总字数 | 插入章节后回填 |
| `audit_task_id` | `VARCHAR(64)` | 否 | 审核任务 ID | 外部导入推荐写 `NULL` |
| `created_at` | `DATETIME(3)` | 否 | 创建时间 | 可不传，走默认值 |
| `updated_at` | `DATETIME(3)` | 否 | 更新时间 | 可不传，走默认值 |

### 5.2 `novel_chapter` 章节表

用途：

- 保存章节目录元数据和章节正文

建表关键约束：

- 主键：`id`
- 外键：`novel_id -> novel.id`
- 唯一索引：`uk_novel_chapter_no (novel_id, chapter_no)`

字段说明：

| 字段 | 类型 | 必填 | 说明 | 外部爬虫写入建议 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT` | 否 | 自增主键 | 不传，让数据库生成 |
| `novel_id` | `BIGINT` | 是 | 所属小说 ID | 必须先有 `novel.id` |
| `chapter_no` | `INT` | 是 | 章节序号 | 推荐从 `1` 开始连续递增 |
| `title` | `VARCHAR(200)` | 是 | 章节标题 | 去首尾空格 |
| `content` | `MEDIUMTEXT` | 是 | 章节正文 | 必须为纯文本或可控文本 |
| `status` | `VARCHAR(32)` | 是 | 章节状态 | 外部导入推荐写 `PUBLISHED` |
| `audit_task_id` | `VARCHAR(64)` | 否 | 审核任务 ID | 外部导入推荐写 `NULL` |
| `published_at` | `DATETIME(3)` | 否 | 发布时间 | 推荐写当前时间或源站发布时间 |
| `created_at` | `DATETIME(3)` | 否 | 创建时间 | 可不传 |
| `updated_at` | `DATETIME(3)` | 否 | 更新时间 | 可不传 |

### 5.3 与爬虫导入强相关但通常不需要写入的表

#### `user_account`

用途：

- 提供 `author_id` 的来源

外部爬虫要求：

- `novel.author_id` 必须引用一个真实存在的用户 ID
- 如果你准备长期导入，建议人工创建一个固定导入用户

#### `user_profile`

用途：

- 仅用于前端展示作者昵称、头像等信息

外部爬虫要求：

- 只要 `author_id` 对应的账号和资料存在即可
- 不建议每次导入时由爬虫修改作者资料

## 6. 状态字段约定

### 6.1 小说状态 `novel.status`

可选值：

- `DRAFT`
- `PENDING_AUDIT`
- `REJECTED`
- `PUBLISHED`
- `ON_SHELF`
- `OFF_SHELF`
- `BANNED`

外部爬虫直连数据库时，推荐使用：

- `ON_SHELF`

原因：

- 阅读模块读取小说时，通常要求小说对外可见
- `ON_SHELF` 表示已经可展示、可阅读

不推荐外部爬虫直接写：

- `BANNED`
- `PENDING_AUDIT`

因为这两个状态涉及平台运营和审核流程。

### 6.2 章节状态 `novel_chapter.status`

可选值：

- `DRAFT`
- `PENDING_AUDIT`
- `REJECTED`
- `PUBLISHED`

外部爬虫直连数据库时，推荐使用：

- `PUBLISHED`

原因：

- 阅读链路通常只展示已发布章节

## 7. 字段映射规则

外部爬虫抓到的内容，建议映射如下：

| 抓取字段 | 入库字段 |
| --- | --- |
| 小说标题 | `novel.title` |
| 小说简介 | `novel.intro` |
| 小说封面链接 | `novel.cover_url` |
| 分类 ID | `novel.category_id` |
| 标签 ID 列表 | `novel.tag_ids` |
| 章节序号 | `novel_chapter.chapter_no` |
| 章节标题 | `novel_chapter.title` |
| 章节正文 | `novel_chapter.content` |
| 源站发布时间 | `novel_chapter.published_at` |

补充规则：

- `tag_ids` 必须是逗号分隔的数字串，例如 `1,2,8`
- 空标签建议写 `''` 或 `NULL`
- `word_count` 建议按“所有已发布章节正文长度之和”计算
- `latest_chapter_id` 建议指向“当前小说最大 `chapter_no` 且状态为 `PUBLISHED` 的章节 ID”

## 8. 推荐导入流程

### 8.1 新小说首次导入

推荐顺序：

1. 确定 `author_id`
2. 插入 `novel`，先把 `latest_chapter_id = NULL`、`word_count = 0`
3. 拿到生成的 `novel.id`
4. 按章节顺序插入 `novel_chapter`
5. 查询最新章节 ID
6. 计算总字数
7. 回写 `novel.latest_chapter_id` 和 `novel.word_count`
8. 将 `novel.status` 设为 `ON_SHELF`

### 8.2 已存在小说增量更新

推荐顺序：

1. 先按 `author_id + title` 查询已有小说
2. 找到 `novel.id`
3. 更新小说基础信息：`intro`、`cover_url`、`category_id`、`tag_ids`
4. 按 `(novel_id, chapter_no)` 判断章节是否已存在
5. 已存在：
   更新 `title`、`content`、`status`、`published_at`
6. 不存在：
   插入新章节
7. 全部章节处理完后，重新计算：
   `latest_chapter_id`
   `word_count`

### 8.3 推荐使用事务

一次导入一本小说时，建议使用数据库事务。

原因：

- 避免只写入小说主表但章节失败
- 避免章节写完但 `word_count` / `latest_chapter_id` 没回填

## 9. 去重与幂等策略

### 9.1 小说去重

当前数据库没有 `author_id + title` 唯一索引，因此外部爬虫必须自己控制。

推荐策略：

- 以 `author_id + title` 作为业务唯一键
- 每次导入前先执行：

```sql
SELECT id
FROM novel
WHERE author_id = ?
  AND title = ?
LIMIT 1;
```

如果查询到结果：

- 走更新流程

如果没有结果：

- 走插入流程

### 9.2 章节去重

章节表已经有唯一索引：

- `uk_novel_chapter_no (novel_id, chapter_no)`

推荐策略：

- 以 `novel_id + chapter_no` 作为章节唯一键

可选实现：

- 先查后更
- 或直接用 MySQL `INSERT ... ON DUPLICATE KEY UPDATE`

## 10. 推荐 SQL 模板

### 10.1 查询小说是否存在

```sql
SELECT id, title, status
FROM novel
WHERE author_id = :authorId
  AND title = :title
LIMIT 1;
```

### 10.2 插入小说主表

```sql
INSERT INTO novel (
    author_id,
    title,
    intro,
    cover_url,
    category_id,
    tag_ids,
    status,
    latest_chapter_id,
    word_count,
    audit_task_id
) VALUES (
    :authorId,
    :title,
    :intro,
    :coverUrl,
    :categoryId,
    :tagIds,
    'ON_SHELF',
    NULL,
    0,
    NULL
);
```

### 10.3 更新小说基础信息

```sql
UPDATE novel
SET intro = :intro,
    cover_url = :coverUrl,
    category_id = :categoryId,
    tag_ids = :tagIds,
    status = 'ON_SHELF',
    audit_task_id = NULL,
    updated_at = CURRENT_TIMESTAMP(3)
WHERE id = :novelId;
```

### 10.4 插入或更新章节

```sql
INSERT INTO novel_chapter (
    novel_id,
    chapter_no,
    title,
    content,
    status,
    audit_task_id,
    published_at
) VALUES (
    :novelId,
    :chapterNo,
    :title,
    :content,
    'PUBLISHED',
    NULL,
    :publishedAt
)
ON DUPLICATE KEY UPDATE
    title = VALUES(title),
    content = VALUES(content),
    status = VALUES(status),
    audit_task_id = VALUES(audit_task_id),
    published_at = VALUES(published_at),
    updated_at = CURRENT_TIMESTAMP(3);
```

### 10.5 回填小说聚合字段

```sql
UPDATE novel
SET latest_chapter_id = (
        SELECT t.id
        FROM (
            SELECT id
            FROM novel_chapter
            WHERE novel_id = :novelId
              AND status = 'PUBLISHED'
            ORDER BY chapter_no DESC, id DESC
            LIMIT 1
        ) t
    ),
    word_count = (
        SELECT COALESCE(SUM(CHAR_LENGTH(content)), 0)
        FROM novel_chapter
        WHERE novel_id = :novelId
          AND status = 'PUBLISHED'
    ),
    updated_at = CURRENT_TIMESTAMP(3)
WHERE id = :novelId;
```

说明：

- `CHAR_LENGTH` 更接近“字符数”
- 如果你希望与项目现有统计逻辑完全一致，可改用 `LENGTH`
- 现有后端代码在统计时使用的是数据库 `length(content)` 逻辑

## 11. 外部爬虫程序推荐数据模型

建议在爬虫程序内部先整理为如下结构，再统一写库：

```json
{
  "authorId": 1002,
  "title": "小说标题",
  "intro": "小说简介",
  "coverUrl": "https://example.com/cover.jpg",
  "categoryId": 1,
  "tagIds": [1, 2, 5],
  "status": "ON_SHELF",
  "chapters": [
    {
      "chapterNo": 1,
      "title": "第一章",
      "content": "正文内容",
      "publishedAt": "2026-03-23 10:00:00.000"
    }
  ]
}
```

入库前转换规则：

- `tagIds: [1,2,5] -> tag_ids: "1,2,5"`
- `chapters` 按 `chapterNo` 升序排序
- 空字符串字段统一做 `trim`

## 12. 数据质量校验清单

每次导入一本小说后，建议至少校验以下内容：

- `novel` 中存在 1 条目标小说记录
- `novel.author_id` 正确
- `novel.title` 不为空
- `novel.status = 'ON_SHELF'`
- `novel.latest_chapter_id` 能在 `novel_chapter` 中查到
- `novel.word_count >= 0`
- `novel_chapter` 中章节数与抓取数一致
- `(novel_id, chapter_no)` 没有重复
- 所有导入章节的 `status = 'PUBLISHED'`

推荐核对 SQL：

```sql
SELECT n.id,
       n.title,
       n.status,
       n.latest_chapter_id,
       n.word_count,
       COUNT(c.id) AS chapter_count
FROM novel n
LEFT JOIN novel_chapter c ON c.novel_id = n.id
WHERE n.id = :novelId
GROUP BY n.id, n.title, n.status, n.latest_chapter_id, n.word_count;
```

## 13. 常见问题

### 13.1 为什么不建议爬虫直接写 `audit_task`

因为 `audit_task` 属于审核子系统数据，直接写入无法保证状态机一致，也没有必要。外部内容导入如果就是“直接上线内容”，只需要正确写 `novel` 和 `novel_chapter`。

### 13.2 为什么 `word_count` 不能随便写一个值

因为前端和后端都会把它当作小说聚合信息使用。如果它与章节正文不一致，会导致排序、展示、统计全部偏差。

### 13.3 为什么 `latest_chapter_id` 不能写错

因为小说详情页和目录页会依赖这个字段定位“最新章节”。如果它指向了错误章节，前端展示会错。

### 13.4 如果源站没有分类和标签怎么办

可以写：

- `category_id = NULL`
- `tag_ids = ''`

### 13.5 如果后续还想走平台审核怎么办

那就不建议外部程序直接改库，而应改成：

1. 先通过业务接口建书、建章
2. 再走审核和上架流程

## 14. 推荐实施方案

如果你的目标是“尽快把外部站点数据导入当前项目并可读”，推荐采用下面的最小方案：

1. 固定导入作者为 `1002`
2. 小说状态统一写 `ON_SHELF`
3. 章节状态统一写 `PUBLISHED`
4. 章节按 `chapter_no` 自增写入
5. 每本小说导入完成后重新计算 `latest_chapter_id` 和 `word_count`
6. 每次导入前按 `author_id + title` 查重
7. 每章导入时按 `(novel_id, chapter_no)` 做更新而不是盲插
