# 阅读模块

## 1. 模块目标

- 为 PC Web/H5 提供稳定、高性能的阅读链路：小说详情、章节目录、章节正文。
- 提供阅读进度、书签与阅读历史的写入与恢复能力，保证幂等与可追溯。
- 在“章节正文大字段 + 热点读取”场景下，通过分层缓存与防护策略控制数据库压力。

## 2. 职责边界

模块负责：

- 小说详情与章节目录的读取用例编排（读取侧，不负责编辑侧写入）。
- 章节正文读取与缓存命中链路（L1/L2/L3 协作，防击穿/穿透）。
- 阅读进度、书签、历史记录的写入与读取。
- 阅读行为基础埋点（阅读次数/最近阅读），仅记录“最小必要数据”，统计聚合可异步化。

模块不负责：

- 小说与章节内容的创建/编辑/发布（由小说管理模块负责）。
- 评论、点赞、收藏、关注等互动（由社区互动模块负责）。
- 推荐列表生成（由推荐引擎模块负责）。
- 认证授权与全站限流框架实现（由安全治理模块负责）；本模块只声明需要的权限与限流粒度。

禁止依赖：

- 禁止跨模块直接写入 `novel`、`novel_chapter` 的发布/状态字段（只读）。
- 禁止直接访问审核、推荐模块内部存储或缓存键，只能通过其公开接口或事件协作。

## 3. 模块结构

建议包结构（以 `com.ainovel.reading` 为根）：

- `controller`
  - `ReadingNovelController`（小说详情、目录）
  - `ReadingChapterController`（章节正文）
  - `ReadingProgressController`（进度/书签/历史）
- `service`
  - `NovelReadService`、`ChapterReadService`、`ProgressService`
  - `ReadAccessGuardService`（反爬校验、频控策略适配）
- `domain`
  - `ReadingProgress`、`Bookmark`、`ReadingHistory`
- `mapper`
  - `ReadingProgressMapper`（如进度单独表）
  - 读取侧可复用 `NovelMapper/ChapterMapper` 的只读查询接口，但不得引入写入方法
- `dto` / `vo` / `converter`
- `cache`
  - 仅放“阅读域”的缓存编排与 Key 规范，不放通用缓存框架实现

## 4. 核心流程

### 4.1 章节目录查询（分页）

1. 校验小说可见性：`novel.status` 必须在可阅读集合（例如 `ON_SHELF`）。
2. 目录读取优先命中缓存 `novel:chapters:{novelId}`（缓存内容为“章节元信息列表”，不含正文）。
3. 未命中则查询数据库（按 `chapter_no` 排序），回填缓存。
4. 返回分页结果：`page/size` 必填，禁止全量返回。

失败与降级：

- Redis 不可用时回源数据库，但必须做接口限流与分页上限保护。

### 4.2 章节正文读取（高频主链路）

读取顺序（Cache Aside）：

1. 先查 L1（可选，仅缓存小对象或短文本片段，不建议缓存完整正文）。
2. 再查 L2 Redis：`chapter:content:{chapterId}`。
3. 未命中：校验 `chapterId` 是否可能存在（布隆过滤器/空值缓存，见 7.3）。
4. 回源数据库读取章节正文（只读），并回填 Redis。
5. 返回给客户端，按需返回“可渲染正文”与最小元数据（标题、字数、发布时间）。

击穿防护：

- 热点章节 Key 使用互斥锁或“逻辑过期”策略，避免并发回源压垮数据库。

### 4.3 阅读进度写入（幂等）

1. 校验登录态（匿名用户可不写进度，只在前端本地存储）。
2. 以 `(userId, novelId)` 为幂等维度更新进度：
   - 新进度的 `updated_at` 必须单调递增。
   - 若上报 `chapterId` 早于历史最新章节，可接受但不得覆盖“最新阅读时间”。
3. 写入 `reading_progress`（或按设计拆表），必要时同步刷新 Redis 近端缓存（可选）。

失败与重试：

- 写入失败返回可重试错误码，客户端可做短周期重试（带幂等键）。

### 4.4 进度/历史恢复

1. 优先读取 Redis `reading:progress:{userId}:{novelId}`（如启用）。
2. 未命中则查询数据库，回填缓存。
3. 返回当前章节与进度百分比/页内偏移（偏移由客户端定义并上报，服务端仅存储数值）。

## 5. 接口约束

通用约束：

- 统一前缀：`/api/v1/**`。
- 统一响应：`Result<T>`。
- 分页：目录与列表接口必须分页，并设置 `size` 上限（例如 `size <= 50`）。
- 幂等：进度写入接口支持 `Idempotency-Key`（请求头），后端以 `userId+novelId+key` 去重（实现细节由安全/基础设施模块补齐）。

### 5.1 小说详情（阅读态）

- 用途：展示小说基础信息与可阅读状态（不含作者编辑态字段）
- 路径：`GET /api/v1/novels/{novelId}`
- 权限：公开可读，但必须通过可见性校验（上架/封禁）
- 关键参数：`novelId`
- 缓存：`novel:detail:{novelId}`

### 5.2 章节目录（分页）

- 用途：分页返回章节元数据列表（不含正文）
- 路径：`GET /api/v1/novels/{novelId}/chapters?page=&size=`
- 权限：公开可读，需可见性校验
- 约束：
  - `page>=1`，`size` 必填且有上限
  - 返回字段必须最小化：`chapterId/chapterNo/title/status/publishedAt/wordCount`
- 缓存：`novel:chapters:{novelId}`

### 5.3 章节正文

- 用途：读取章节正文
- 路径：`GET /api/v1/chapters/{chapterId}/content`
- 权限：公开可读，需小说与章节均处于“可阅读”状态
- 反爬约束：
  - 必须应用接口级限流（IP/用户/设备）
  - 可选：要求携带短期阅读票据（后续由安全治理模块定义）
- 缓存：`chapter:content:{chapterId}`

### 5.4 写入阅读进度

- 用途：上报/更新阅读进度、最后阅读章节
- 路径：`POST /api/v1/reading-progress`
- 权限：登录用户
- 幂等：必须支持 `Idempotency-Key`
- 关键参数（示意）：
  - `novelId`、`chapterId`、`progressPercent`、`clientTs`、`deviceId`（可选）
- 约束：
  - `progressPercent` 范围 `[0,100]`
  - `clientTs` 不可信，仅用于参考，服务端以 `updated_at` 为准

## 6. 数据约束

### 6.1 阅读侧只读数据约束

- `novel`：必须过滤不可见状态（例如 `OFF_SHELF`、`BANNED`）。
- `novel_chapter`：
  - 目录只返回 `PUBLISHED` 章节（或由小说管理模块定义“可读章节集合”）。
  - 正文读取必须校验章节状态可读。

### 6.2 阅读进度数据约束（`reading_progress`）

建议表结构（与总方案一致）：

- 唯一性：建议对 `(user_id, novel_id)` 建唯一约束或在业务层保证唯一。
- 字段约束：
  - `chapter_id` 必须属于 `novel_id`（写入时校验）。
  - `progress_percent` 范围 `[0,100]`。
- 更新策略：
  - `updated_at` 必须由服务端写入，用于判定“最新进度”。
  - 允许写入更早章节，但不允许倒退覆盖“最近阅读时间”的语义（由业务规则约束）。

## 7. 缓存、事务与一致性约束

### 7.1 缓存键规范（阅读域）

- 小说详情：`novel:detail:{novelId}`
- 章节目录：`novel:chapters:{novelId}`
- 章节正文：`chapter:content:{chapterId}`
- 进度缓存（可选）：`reading:progress:{userId}:{novelId}`

Key 规范：

- 统一小写前缀与冒号分隔，禁止在 key 中拼接未清洗的用户输入。
- TTL 必须带随机抖动（例如 `baseTtl + random(0..baseTtl*0.1)`）以降低雪崩风险。

### 7.2 一致性策略

- 采用 `Cache Aside`：
  - 读：缓存未命中回源数据库再回填。
  - 写（进度）：以数据库为准，缓存可选写穿或删缓存后回填。
- 小说/章节发布或下架导致的缓存失效：
  - 由小说管理模块负责写后失效 `novel:detail`、`novel:chapters`。
  - 章节正文缓存 `chapter:content` 可由阅读模块在读取时自然过期，也可由“发布事件”触发失效（后续基础设施定义事件机制）。

### 7.3 穿透、击穿、防雪崩

- 穿透：
  - 使用布隆过滤器（小说/章节主键）或空值缓存（短 TTL）防止扫描不存在 ID。
- 击穿：
  - 热点章节正文使用互斥锁或逻辑过期防并发回源。
- 雪崩：
  - TTL 抖动，批量预热受控，避免统一过期。

### 7.4 事务约束

- 阅读模块以读为主，除进度写入外不应开启长事务。
- 进度写入使用短事务，禁止在事务内进行外部调用（如推荐/审核）。

## 8. 安全约束

- 可见性校验是阅读模块必须执行的安全规则：封禁/下架内容不可读。
- 反爬与接口风控：
  - 章节正文接口必须配置更严格的限流阈值。
  - 对异常访问模式记录风控日志并触发黑名单/挑战（具体实现归安全治理模块）。
- 输入安全：
  - `chapterId/novelId/page/size` 必须做参数校验。
  - 正文返回必须遵守“最小必要字段”，不返回作者内部编辑态信息。

## 9. 异常、日志与审计约束

日志要求：

- 所有阅读链路日志必须包含：`traceId`、`userId(可空)`、`deviceId(可空)`、`ip`、`novelId/chapterId`、耗时、缓存命中层级（L1/L2/DB）。
- 章节正文接口需额外记录：限流判定结果、命中风控策略 ID（如有）。

异常处理：

- 参数错误：`1000-1999`
- 权限/可见性错误：`2000-2999`（包含“内容不可见”）
- 业务状态错误：`3000-3999`（如“章节未发布不可读”）
- 系统错误：`5000-5999`，必须记录堆栈与 `traceId`

告警建议（由基础设施模块落地）：

- 章节正文 DB 回源率异常升高
- 单 IP/单设备读取频率异常
- Redis 命中率持续低于阈值

## 10. 实现边界

- 本模块只定义阅读域的接口契约、缓存编排策略与数据约束。
- 不实现通用缓存组件、分布式锁、限流中间件本体，这些归缓存模块/安全治理模块/基础设施模块。
- 不引入复杂推荐或内容审核逻辑，阅读模块只依赖“内容可见性”与“可读状态”。

## 11. 开发顺序

1. 先实现小说详情与目录分页读取（只读），补齐可见性校验与分页约束。
2. 实现章节正文读取链路：Redis 缓存 + DB 回源 + 基础穿透防护（空值缓存/布隆占位）。
3. 增加击穿防护（互斥锁/逻辑过期）与缓存命中统计日志字段。
4. 实现阅读进度写入与恢复（幂等、短事务、数据约束）。
5. 接入安全治理模块的细粒度限流与风控策略，对章节正文接口做强化。
6. 增加监控指标与告警阈值（命中率、回源率、P95 延迟）。
