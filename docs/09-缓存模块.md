# 缓存模块

## 1. 模块目标

- 提供平台统一的缓存分层与治理能力，覆盖热点读场景（小说详情、章节目录、章节正文、推荐列表、计数类数据、会话态数据）。
- 通过 `L1(Caffeine) + L2(Redis) + L3(布隆/空值/限流)` 组合，降低 MySQL 压力并抵御穿透、击穿、雪崩。
- 为业务模块提供可执行的 Key 规范、TTL 规范、失效策略、一致性策略与监控指标，避免“各写各的缓存”导致系统不可控。

## 2. 职责边界

模块负责：

- 缓存分层设计与统一策略：Key 命名、TTL、序列化、批量加载、失效与预热、热点保护。
- 通用能力：分布式锁（用于热点 Key 击穿保护）、逻辑过期（可选）、布隆过滤器（可选）、空值缓存策略。
- 监控与告警：命中率、加载耗时、回源比率、热点 Key、Redis 连接与慢查询。

模块不负责：

- 业务对象的状态机与权限校验（由各业务模块负责）。
- 业务缓存内容的具体字段裁剪（由业务模块定义 VO，并遵循“返回字段最小化”）。
- 审核/推荐算法与任务调度（由审核引擎/推荐引擎/基础设施模块负责）。

禁止行为：

- 业务模块不得绕过缓存模块规范随意使用临时 Key（临时 Key 必须登记到“Key 规范”章节）。
- 不允许在缓存中长期存储敏感原文（章节正文缓存除外，但必须脱敏日志、限制 TTL、限制访问）。

## 3. 模块结构

建议包结构（以 `com.ainovel.cache` 为根）：

- `config`：Caffeine/Redis 配置、序列化策略、连接池参数。
- `key`：Key 生成器与 Key 常量（统一命名和参数校验）。
- `l1`：本地缓存包装（Caffeine Cache）。
- `l2`：Redis 访问封装（`RedisTemplate`/`StringRedisTemplate` 使用约束）。
- `guard`：穿透/击穿/雪崩防护（布隆、空值、互斥锁、随机 TTL）。
- `metrics`：命中率、回源统计、热点 Key 采样、告警接口（对接基础设施监控）。
- `support`：分布式锁（`lock:{key}`）、幂等辅助、批量加载工具。

## 4. 核心设计

## 4.1 L1 本地缓存（Caffeine）

定位：同 JVM 内热点小对象缓存，减少 Redis 往返。

适用数据：

- 字典类：分类、标签、权限点、配置项。
- 热点摘要：小说详情（不含正文）、推荐结果摘要（小对象）、热点榜单摘要。

约束：

- 只缓存小对象与可序列化对象，禁止缓存章节正文大文本。
- TTL 建议：`30s ~ 5min`，必须允许快速失效以避免跨实例不一致。
- L1 仅作为“加速层”，不能作为唯一真相来源。

## 4.2 L2 分布式缓存（Redis）

定位：跨实例共享缓存与会话态缓存，承担主要读写缓冲。

适用数据：

- 内容类：小说详情、章节目录、章节正文（按章）、推荐列表。
- 会话类：Token 黑名单、刷新凭证、`login_version`。
- 计数类：评论/点赞/收藏计数的写缓冲（最终一致性）。

约束：

- TTL 必须显式设置；禁止永久 Key（除非明确为“配置项”并有运维手段）。
- 存储结构选择：
  - 大对象优先使用 `String(JSON)` 或 `Hash`（字段级更新场景）。
  - 计数类使用 `String(INCR)` 或 `Hash`（按维度计数）。
- 热点 Key 必须启用击穿保护（互斥锁或逻辑过期）。

## 4.3 L3 防护层（穿透/击穿/雪崩）

### 4.3.1 防穿透

- 布隆过滤器：
  - 适用：`novelId/chapterId/userId` 等高频主键查询。
  - 规则：查不到且布隆判定“不存在”则直接返回 404 类业务错误，不打数据库。
- 空值缓存：
  - 适用：布隆未覆盖或存在短期不一致的对象。
  - TTL：`30s ~ 120s`（短 TTL，避免误伤）。
  - 值：使用固定占位符（如 `"__NULL__"`），避免与合法值冲突。

### 4.3.2 防击穿

适用：热点 Key 过期瞬间大量请求回源数据库。

方案二选一（初期优先互斥锁）：

- 互斥锁（推荐）：
  - `lock:{cacheKey}`，短 TTL（`3s~10s`）防死锁。
  - 只有拿到锁的线程回源 DB 并回填缓存，其他线程短暂等待或快速失败重试。
- 逻辑过期（可选）：
  - 缓存值携带 `expireAt`，过期后仍返回旧值，同时触发后台刷新。
  - 适用：对强一致要求不高的展示类数据（榜单、推荐摘要）。

### 4.3.3 防雪崩

- TTL 随机抖动：`ttl = baseTtl + random(0, jitter)`，对同类 Key 统一抖动窗口。
- 分批预热：发布/上架等大事件触发时只预热 TopN 热点，不做全量预热。

## 5. 缓存键规范（Key Naming）

统一规则：

- 以业务域作为前缀：`auth:`、`user:`、`novel:`、`chapter:`、`comment:`、`recommend:`、`audit:`。
- 固定层级分隔符 `:`，参数部分使用 `{}` 表示变量。
- Key 必须可逆定位业务对象（便于排查），禁止写入 PII 明文（如手机号、邮箱）。

核心 Key 清单（示例）：

- `auth:token_blacklist:{jti}`：JWT 黑名单（TTL=token 剩余有效期）
- `auth:login_version:{userId}`：强制下线版本号（TTL 可设置为较长，如 30d）
- `user:profile:{userId}`：用户资料（TTL 10min + jitter）
- `novel:detail:{novelId}`：小说详情（不含正文）（TTL 5min + jitter）
- `novel:chapters:{novelId}`：章节目录（TTL 5min + jitter）
- `chapter:content:{chapterId}`：章节正文（TTL 30min + jitter，或按热度动态）
- `comment:list:{targetType}:{targetId}:{page}:{size}`：评论列表页缓存（TTL 30s~2min）
- `comment:count:{targetType}:{targetId}`：评论数（TTL 1~5min 或计数写缓冲）
- `recommend:home:{userId}`：首页推荐（TTL 5~30min，随策略调整）
- `bloom:novel`、`bloom:chapter`：布隆过滤器数据结构（由实现决定具体存储方式）

约束：

- 分页列表 Key 必须包含分页参数，禁止“只缓存第一页然后误用”。
- 章节正文 Key 禁止包含用户维度（除非做付费/权限差异化内容，当前不需要）。

## 6. 失效与更新流程

统一模式：`Cache Aside Pattern`。

### 6.1 读流程（通用）

1. 先查 L1（若该 Key 配置允许）。
2. 未命中查 L2（Redis）。
3. 仍未命中：
  - 先执行 L3 防护（布隆/空值/互斥锁）。
  - 通过后回源 DB。
4. 回填 L2（必要时回填 L1）。
5. 返回结果。

### 6.2 写流程（通用）

1. 写 DB（事务内）。
2. 事务提交后：
  - 删除缓存（优先）或更新缓存（若可精确更新）。
  - 对热点数据可触发异步预热（可选）。

约束：

- 禁止“先删缓存再写 DB”，避免并发下出现脏回填。
- 删除缓存优先于更新缓存，除非能保证更新字段与读模型完全一致。

### 6.3 典型场景失效清单（示例）

- 小说基础信息更新：
  - 删除 `novel:detail:{novelId}`
- 章节新增/编辑（草稿）：
  - 删除 `novel:chapters:{novelId}`
- 章节发布：
  - 删除 `novel:chapters:{novelId}`
  - 删除 `chapter:content:{chapterId}`（若存在旧缓存）
- 评论新增/删除：
  - 删除 `comment:list:*`（按目标维度精准失效，至少失效相关 target 的前几页）
  - 更新或删除 `comment:count:{targetType}:{targetId}`

## 7. 一致性约束

一致性等级定义：

- 强一致（读写必须一致）：登录态、权限相关、封禁状态。
- 最终一致：计数类（点赞/收藏/评论数）、推荐列表、榜单。
- 允许短暂不一致：展示类摘要（昵称、头像、简介展示）。

约束：

- 强一致数据禁止仅靠缓存：必须有 DB 真相，并对关键操作做二次校验。
- 计数类采用“Redis 计数缓冲 + 异步落库”时：
  - 必须保证落库幂等（按事件 ID 或时间窗口聚合）。
  - Redis 与 DB 的偏差必须可观测（监控指标）。
- 发布/审核回写等状态机关键路径：
  - 缓存只加速读取，不参与状态机判定。

## 8. 性能约束

- 章节正文读取必须做到“绝大多数命中 L2”，数据库回源应受控（击穿保护）。
- 单接口缓存交互次数上限建议：
  - 读：L1 0~1 次 + L2 0~1 次 + 锁 0~1 次
  - 写：删除/更新 Key 次数应可枚举，禁止无边界批量删 Key
- Redis 操作优先 pipeline/批量（目录/列表类），避免 N+1。

## 9. 异常、监控与日志约束

日志字段（结构化）最小集合：

- `traceId`、`cacheKey`（可脱敏或 hash）、`cacheOp`（GET/SET/DEL/LOCK）、`hit`（L1/L2/MISS）、`elapsedMs`、`fallback`（DB/NULL/BLOOM_BLOCK）、`exception`（可选）。

监控指标建议：

- L1 命中率、L2 命中率、回源 DB 比率。
- 热点 Key TopN（采样）。
- Redis QPS、连接数、命令耗时分位（P95/P99）、慢命令数。
- 锁等待次数、锁超时次数、空值缓存命中次数、布隆拦截次数。

告警建议：

- 命中率突降且 DB QPS 上升。
- 锁超时/等待激增（可能出现击穿或死锁风险）。
- Redis 慢命令或连接池耗尽。

## 10. 实现边界

- 本模块定义统一规范与通用工具，业务模块必须通过约定的 Key 生成与缓存访问封装使用缓存。
- 初期不引入复杂的“多级缓存一致性广播”（如 pub/sub 广播失效），以“写后删缓存 + 短 TTL”控制复杂度；后续有明确需求再演进。
- 布隆过滤器与分布式锁属于增强能力，必须可降级：布隆不可用时退回空值缓存；锁不可用时采用限流与快速失败保护 DB。

## 11. 开发顺序

1. 建立 Key 命名规范与 Key 生成器，落地 `cache:key` 清单并约束评审。
2. 接入 Caffeine 与 Redis 基础配置，确定序列化策略与基础监控埋点。
3. 实现通用 `Cache Aside` 访问封装：读穿、回填、删除、随机 TTL 抖动。
4. 增加 L3 防护：空值缓存与互斥锁；随后引入布隆过滤器（可选）。
5. 为阅读链路（章节正文）与登录态链路（token/version）优先落地缓存治理。
6. 完成指标与告警对接，补齐热点 Key 采样与问题排查手册（可放在 `docs/` 追加）。
