# 审核引擎模块

## 1. 模块目标

- 为平台提供内容合规能力，对小说简介、章节正文、评论等文本进行自动审核与人工复审闭环。
- 为上游业务（小说管理、社区互动）提供稳定的“受理审核、查询结果、回写结论”接口，保证幂等、可追溯与最终一致性。
- 通过异步处理与队列化削峰，避免审核链路阻塞核心写入链路。

## 2. 职责边界

模块负责：

- 审核任务（`audit_task`）的创建、状态机流转、结果持久化与审计记录。
- 自动审核：敏感词、规则匹配、频率/模式检测（规则版起步）。
- 人工复审：任务分配、处理结论、驳回原因记录、复审审计。
- 结果回写：向业务模块提供可拉取结果的查询接口，并预留回调/事件推送能力。

模块不负责：

- 上游业务的状态机与发布逻辑（小说/章节发布仍由小说管理模块执行）。
- 统一鉴权与限流的实现细节（由安全治理模块负责）；本模块只声明权限点要求。
- 推荐与内容分发逻辑。

禁止依赖：

- 不允许直接读写上游业务表（如 `novel`、`novel_chapter`、`comment`）完成状态变更。
- 不允许在审核模块内部实现“业务发布/下架”等动作，仅返回结论与原因摘要。

## 3. 模块结构

建议包结构（以 `com.ainovel.audit` 为根）：

- `controller`：对外审核 API（管理端 + 内部回写/查询）。
- `service`：用例层（创建任务、执行自动审核、复审编排、结果查询）。
- `domain`：领域对象与状态机（`AuditTask`, `AuditStatus`, `RiskLevel`, `BizType`）。
- `engine`：审核执行器（规则引擎、敏感词匹配器、策略编排）。
- `repository/mapper`：MyBatis Mapper（`AuditTaskMapper`, `AuditRuleMapper` 可选）。
- `dto`：请求 DTO（提交审核、复审结论、查询条件）。
- `vo`：返回 VO（任务详情、任务列表、审核结果）。
- `converter`：DTO/DO/VO 转换。

## 4. 核心流程

### 4.1 自动审核主流程（异步）

适用对象：

- `BIZ_NOVEL_INTRO`：小说简介
- `BIZ_CHAPTER_CONTENT`：章节正文
- `BIZ_COMMENT`：评论/回复文本

流程：

1. 上游提交审核请求（同步返回“已受理”）。
2. 审核模块创建 `audit_task`：
  - 保存内容快照（`content_snapshot`）与内容哈希（`content_hash`）。
  - 初始化状态为 `PENDING`（待处理）。
3. 投递异步执行（`@Async` 或任务队列，后续可接 MQ）。
4. 自动审核执行器处理：
  - 基础清洗：去控制字符、归一化空白、截断超长。
  - 规则检测：敏感词、正则模式、黑名单片段、重复刷屏模式（规则版）。
  - 风险分级：`LOW/MEDIUM/HIGH`。
5. 结论落库：
  - `PASS`：通过，可放行。
  - `REJECT`：驳回，上游需提示修改。
  - `MANUAL_REVIEW`：进入人工复审池。
6. 上游通过“查询审核结果”获取结论并更新自身状态机；或后续采用回调/事件订阅方式推送结论。

约束：

- 自动审核必须幂等：同一 `biz_type + biz_id + content_hash` 在未终结前重复提交返回同一 `taskId`。
- 自动审核不得阻塞上游写入：提交接口只做校验与任务创建，不执行耗时检测。

### 4.2 人工复审流程

触发场景：

- 自动审核结论为 `MANUAL_REVIEW`。
- 运营策略要求某些内容类型强制人工（可配置）。

流程：

1. 管理端获取待复审任务列表（按风险等级、类型、时间排序）。
2. 领取任务（可选）或直接处理：
  - 校验权限：`ROLE_ADMIN` 或 `ROLE_REVIEWER`。
  - 写入 `reviewer_id` 与领取时间（可选字段）。
3. 提交复审结论：
  - `PASS` 或 `REJECT`。
  - 驳回必须填写 `reject_reason_code` 与 `reject_reason_text`（文本需脱敏）。
4. 状态流转为终态并记录审计日志。

约束：

- 复审提交必须幂等：同一 `taskId` 在终态后重复提交应返回“已处理”并拒绝再次变更。
- 并发控制：同一任务同一时间只能被一个复审者处理（数据库乐观锁或状态机校验）。

## 5. 接口约束

通用约束：

- 统一前缀：`/api/v1/**`。
- 统一响应：`Result<T>`。
- 所有接口必须记录 `traceId`，并将 `taskId/bizType/bizId` 写入结构化日志字段。
- 提交类接口必须支持幂等（`Idempotency-Key` 或内容哈希）。

### 5.1 受理审核

- 接口名称：提交文本审核
- 请求方式：`POST`
- 请求路径：`/api/v1/audit/text`
- 权限：业务端服务调用或具备相应角色（由安全治理模块配置具体策略）
- 请求参数（示例）：
  - `bizType`（string，必填）：`BIZ_NOVEL_INTRO`/`BIZ_CHAPTER_CONTENT`/`BIZ_COMMENT`
  - `bizId`（long，必填）：上游对象 ID
  - `content`（string，必填）：文本内容
  - `operatorId`（long，可选）：提交人（作者/用户）
  - `source`（string，可选）：来源（author/app/admin）
- 响应数据：
  - `taskId`（long）：审核任务 ID
  - `status`（string）：`PENDING`

约束：

- 内容长度限制：按业务类型配置上限；超出直接拒绝（参数错误）或截断并标记风险（需明确策略）。
- 禁止返回敏感命中细节，只返回受理信息。

### 5.2 查询审核结果（上游拉取）

- 请求方式：`GET`
- 请求路径：`/api/v1/audit/tasks/{taskId}`
- 权限：上游服务或管理端
- 响应字段：
  - `auditStatus`：`PASS/REJECT/MANUAL_REVIEW/PENDING`
  - `riskLevel`：`LOW/MEDIUM/HIGH`
  - `reasonCode`、`reasonText`：原因摘要（可为空）
  - `reviewerId`、`reviewedAt`（如已复审）

约束：

- 上游不得基于 `reasonText` 推导具体敏感词规则；`reasonText` 必须为对用户可展示的摘要。

### 5.3 管理端任务列表与复审

- 任务列表：
  - `GET /api/v1/admin/audit/tasks?status=&bizType=&riskLevel=&page=&size=`
  - 权限：`ROLE_ADMIN`/`ROLE_REVIEWER`
  - 必须分页，默认按 `created_at desc`
- 提交复审结论：
  - `POST /api/v1/admin/audit/tasks/{taskId}/review`
  - 权限：`ROLE_ADMIN`/`ROLE_REVIEWER`
  - 入参：`decision`（PASS/REJECT），`rejectReasonCode`（REJECT 必填），`rejectReasonText`（可选）

## 6. 数据约束（审核任务数据约束）

核心表：`audit_task`（详见总方案）

关键字段约束建议：

- `biz_type` + `biz_id` 必填，表示审核对象。
- `content_snapshot` 必填，保存提交时文本快照，用于追溯与申诉。
- `content_hash` 必填，建议为 `SHA-256`，用于幂等与重复提交识别。
- `audit_status` 状态机建议：
  - `PENDING` -> `PASS`
  - `PENDING` -> `REJECT`
  - `PENDING` -> `MANUAL_REVIEW` -> (`PASS` | `REJECT`)
- `risk_level`：`LOW/MEDIUM/HIGH`（可用 tinyint 存储）。
- `reviewer_id`：仅在人工复审处理时写入。

唯一约束与索引建议：

- 唯一：`uk_biz_content`：`(biz_type, biz_id, content_hash)`（允许同内容只生成一个有效任务）。
- 索引：
  - `idx_status_created_at`：`(audit_status, created_at)`
  - `idx_biz_type_biz_id`：`(biz_type, biz_id)`
  - `idx_risk_level_created_at`：`(risk_level, created_at)`

数据保留策略：

- `content_snapshot` 属于敏感数据，需设置保留期与脱敏策略（至少区分生产/非生产环境）。

## 7. 异步处理约束

执行模型：

- 受理接口同步创建任务并快速返回。
- 自动审核通过异步执行器处理（`@Async` 或任务调度），并限制并发度与队列长度。

重试与失败处理：

- 审核执行失败应将任务标记为 `PENDING` 并记录 `retry_count`（建议字段）与最后错误摘要（不记录原文）。
- 超过最大重试次数后转 `MANUAL_REVIEW` 或 `FAILED`（建议新增状态，若新增需同步到状态机约束）。

幂等与去重：

- 异步执行器按 `taskId` 处理必须幂等，重复投递不应重复落库或覆盖终态。
- 同一 `biz_type + biz_id + content_hash` 不得并发执行两次。

## 8. 安全与审计约束

权限控制（与安全治理模块协作）：

- 管理端接口要求 `ROLE_ADMIN` 或 `ROLE_REVIEWER`。
- 上游服务调用可使用服务间鉴权（后续实现），或通过内网访问 + 签名校验方案（仅设计层声明）。

数据安全：

- `content_snapshot`、`reasonText` 可能包含敏感内容，日志中禁止输出原文。
- 审核规则与敏感词库属于高敏资产，不对外暴露，不写入可读日志。

审计事件（必须落库或结构化日志）：

- `AUDIT_TASK_CREATED`：创建任务
- `AUDIT_AUTO_DECIDED`：自动审核出结论
- `AUDIT_MANUAL_ASSIGNED`（可选）：领取/分配复审
- `AUDIT_MANUAL_DECIDED`：人工复审出结论

审计字段最小集合：

- `traceId`、`taskId`、`bizType`、`bizId`、`operatorId`、`action`、`fromStatus`、`toStatus`、`riskLevel`、`reviewerId`（可选）、`elapsedMs`

## 9. 异常、日志与审计约束

错误码分层（与全局约束一致）：

- `1000-1999`：参数错误（bizType 不合法、content 为空、长度超限）。
- `2000-2999`：权限不足（无复审权限、非管理端访问）。
- `3000-3999`：状态错误（重复复审、终态不可变更）。
- `5000-5999`：系统错误（执行器异常、存储异常）。

日志要求：

- 受理接口记录：`bizType/bizId/contentHash/taskId/operatorId`（不记录内容）。
- 自动审核记录：规则版本（`rule_version`，建议字段）、风险等级、结论、耗时（不记录命中细节）。
- 人工复审记录：复审人、结论、原因码（原因文本只记录可公开摘要）。

告警建议：

- `PENDING` 积压超过阈值（按 bizType 分组）告警。
- 自动审核失败率与重试次数异常告警。
- 人工复审 SLA 超时告警。

## 10. 实现边界

- 本模块仅定义审核任务与结论，不直接修改上游业务状态。
- 自动审核先以规则版实现，避免引入重算法或外部模型调用带来的不可控成本；后续如引入外部审核服务，必须保留本地可回退策略。
- 回写方式默认“上游拉取”，回调/事件推送作为增强能力预留，不作为初期必需依赖。

## 11. 开发顺序

1. 定义 `audit_task` 状态机、基础 DTO/VO、错误码与审计日志字段规范。
2. 实现提交审核受理接口：参数校验、幂等去重（`content_hash`）、任务创建。
3. 实现自动审核执行器（规则版）：敏感词 + 正则 + 风险分级，异步执行与并发控制。
4. 实现结果查询接口：按 `taskId` 与按 `bizType+bizId` 查询（后者可选）。
5. 实现管理端复审能力：任务列表分页、复审提交、并发控制与审计记录。
6. 补齐监控与告警：积压、失败率、耗时分布；完善数据脱敏与保留策略。
