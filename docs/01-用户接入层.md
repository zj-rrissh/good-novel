# 用户接入层

## 1. 模块目标

- 为 PC Web 与移动 H5 提供统一、版本化的 HTTP/HTTPS 接入入口。
- 为通知态/轻量实时能力预留 WebSocket 接入点，并定义降级到轮询的契约。
- 统一对外 API 路径、响应封装 `Result<T>`、错误码分层与必要请求头，降低客户端适配成本与排障成本。

## 2. 职责边界

职责包含：

- 定义 API 版本前缀、路径域（app/admin/ws）与接口契约。
- 定义客户端请求头、鉴权令牌携带方式、幂等键使用规则。
- 定义接入层必须输出的访问日志与审计日志字段规范。

不包含：

- 认证授权实现、权限模型、Token 签发与校验（归属安全治理模块）。
- SSL 终止、静态资源分发、CDN、Nginx 反向代理与边缘限流（归属流量治理/接入网关）。
- 任何业务规则与数据写入逻辑（归属各业务模块）。

依赖：

- 依赖安全治理模块：鉴权、限流、风控、XSS/输入清洗策略落地。
- 依赖基础设施模块：TraceId/RequestId 生成、统一异常与统一响应落地、日志采集。

## 3. 模块结构

以“路径域 + 协议”划分接入入口：

- PC Web（HTTP/HTTPS）：页面与后台管理/创作入口，走 `/api/admin/v1/**` 与 `/api/v1/**`。
- 移动 H5（HTTP/HTTPS）：阅读与互动入口，走 `/api/v1/**`。
- WebSocket（WSS）：通知与轻量实时入口，走 `/ws/v1/**`。

路径规范（必须）：

- 普通用户：`/api/v1/**`
- 管理后台：`/api/admin/v1/**`
- WebSocket：`/ws/v1/**`（Nginx 侧可配置 `/ws/` 升级）

## 4. 核心流程

### 4.1 PC Web（HTTP/HTTPS）

1. 静态资源优先由 CDN/Nginx 提供，业务接口仅承载数据。
2. 对需要登录的接口，客户端必须携带 `Authorization: Bearer <accessToken>`。
3. 所有响应使用 `Result<T>` 返回；客户端只依赖 `code/message/data`。
4. Token 过期时按刷新契约获取新 token（刷新机制实现归属安全治理模块）。

### 4.2 移动 H5（HTTP/HTTPS）

1. 阅读页接口必须“按需加载”：目录分页、正文按章加载。
2. 弱网下优先调用摘要/目录接口，正文接口允许重试但必须受限流约束。
3. 阅读进度上报采用“允许最终一致”的策略（具体落库归属阅读模块）。

### 4.3 WebSocket（WSS）

适用场景（仅限）：

- 系统通知/互动通知/审核通知推送。
- 轻量事件提示（例如“有新回复”），不承诺强一致实时数值。

连接与鉴权约束：

1. 客户端连接 `wss://<host>/ws/v1/connect`。
2. 必须完成鉴权：建议在连接建立后发送 `AUTH` 消息携带短期 token（或在握手阶段携带 token，具体与安全治理模块对齐）。
3. 客户端必须实现心跳与断线重连。

降级策略（必须）：

- WebSocket 不可用时自动切换为 HTTP 轮询拉取消息列表，轮询间隔必须受限流策略约束。

## 5. 接口约束

### 5.1 统一响应

统一响应结构（必须）：

```json
{ "code": 0, "message": "success", "data": {} }
```

统一要求：

- `message` 对外不得泄漏内部堆栈、SQL、主机等敏感信息。
- 列表接口必须分页，禁止无上限返回。

### 5.2 API 版本与兼容性

- 默认只在 `/api/v1` 内做向后兼容变更。
- 破坏性变更必须升版本（例如 `/api/v2`），不得复用旧路径。

### 5.3 必要请求头

- `X-Client`（必填）：`pc-web` 或 `h5` 或其他已登记值。
- `X-App-Version`（建议）：客户端版本号，用于灰度与排障。
- `Idempotency-Key`（按需）：写接口若要求幂等，必须携带。

### 5.4 幂等性

- 适用接口：注册、评论发布、点赞/收藏切换、提交审核等可能被重复点击/重试的写操作。
- 幂等范围建议：`用户维度 + 路径 + Idempotency-Key`，TTL 不少于 5 分钟（与缓存/安全治理模块对齐）。

### 5.5 分域与权限

- `/api/v1/**` 面向普通用户；权限校验由安全治理模块执行。
- `/api/admin/v1/**` 仅面向后台角色；必须更严格限流与审计。

## 6. 数据约束

接入层不定义业务表，但对输入输出做硬约束：

- JSON 编码统一 `UTF-8`。
- 文本类字段必须有最大长度：标题/昵称/简介/评论/章节正文分别设定上限（具体数值由业务模块文档确定，但接入层必须拒绝超长请求）。
- 富文本/用户输入展示字段必须经过 XSS 清洗或白名单渲染策略（实现归属安全治理模块）。
- 文件上传（封面/头像/插图）必须限制类型与大小，并只返回可访问 URL，不返回二进制内容。

## 7. 缓存、事务与一致性约束

- 接入层不直接操作缓存/事务，但必须保证接口设计“可缓存、可限流、可审计”：
  - 热点读接口参数必须稳定（例如 `novelId/chapterId`），便于缓存命中。
  - 列表接口必须有确定排序与分页，否则缓存不可复用。
- 计数类字段（点赞数/评论数）允许最终一致，客户端不得假设强一致。

## 8. 安全约束

- 强制 HTTPS；WebSocket 强制 WSS。
- CORS 必须白名单化，禁止 `*`。
- CSRF 策略与登录态方式一致：若存在 Cookie 会话必须启用 CSRF 防护；若纯 Bearer Token 可按安全治理建议配置。
- 阅读相关接口必须配合限流防爬（IP/用户/设备维度）。
- 登录相关接口必须支持风控升级（例如验证码触发），接入层只定义契约字段，不实现策略。

## 9. 异常、日志与审计约束

### 9.1 错误码分层

- `1000-1999` 参数与请求错误
- `2000-2999` 认证授权错误
- `3000-3999` 业务状态错误
- `5000-5999` 系统内部错误

### 9.2 访问日志（必须字段）

- `traceId`/`requestId`
- `X-Client`、`X-App-Version`（若有）
- `path`、`method`
- `httpStatus`、`code`
- `latencyMs`
- `userId`（可解析时）
- `ip`（按合规处理）
- 对热点读接口建议记录 `cacheHit`

### 9.3 审计日志（必须覆盖）

- 登录、登出、刷新 token
- 发布章节、提交审核、删除内容
- 评论发布/删除、关注/取关、点赞/取消点赞、收藏/取消收藏

## 10. 实现边界

- Controller 层不得承载业务规则，不得直接访问 DB/Redis。
- 接入层只定义 WebSocket 协议与路由边界，不在此层承诺强一致实时数据。
- 灰度/AB 能力只通过请求头或明确参数传递，禁止客户端硬编码环境逻辑。

## 11. 开发顺序

1. 固化路径域与版本前缀：`/api/v1`、`/api/admin/v1`、`/ws/v1`。
2. 定义并实现统一响应 `Result<T>` 与错误码分层（公共模块）。
3. 定义并校验必要请求头：`X-Client`、`X-App-Version`、`Idempotency-Key`。
4. 与安全治理模块对齐鉴权与限流契约，明确 WebSocket 鉴权方式与降级轮询策略。
5. 与基础设施模块对齐 TraceId、访问日志与审计日志字段规范。
