# 持久化模块

## 1. 模块目标

- 为智能小说平台提供稳定、可演进的持久化能力：业务数据（MySQL）与文件资源（封面/头像/插图等）的统一落地与访问约束。
- 提供跨业务模块一致的“表设计、索引、事务边界、数据一致性、演进与迁移”规范，避免后续功能叠加导致数据层失控。
- 在单体阶段优先保证可用性与可维护性，同时预留分库分表、读写分离、冷热分层等扩展空间。

## 2. 职责边界

模块负责：

- MySQL 表设计规范、索引策略、主键策略、时间字段与软删除策略。
- 事务边界约束（哪些写链路必须事务化、哪些链路只保证最终一致）。
- 数据一致性与并发控制原则（幂等、乐观锁、唯一约束、状态机约束）。
- 文件存储的抽象与安全约束（上传/访问/回收/防盗链等）。
- 数据迁移与版本管理要求（Schema 演进策略）。

模块不负责：

- 具体业务用例实现（由各业务模块 service 层负责）。
- 缓存命中策略与缓存一致性实现细节（由缓存模块负责；本模块只声明“写后必须失效/回填”的数据约束）。
- 统一鉴权与限流实现（由安全治理模块负责）。

禁止依赖：

- 业务模块不得跨域直接写入非本模块声明的“非本域表”（例如：小说模块不得写入 `user_message`）。

## 3. 模块结构

建议结构（目标态，后续实现时落到代码包结构）：

- `com.ainovel.persistence`
  - `schema`：表结构设计与变更说明（文档/迁移脚本约定）
  - `model`：通用 DO/枚举（如状态枚举、审计字段）
  - `mapper`：通用 Mapper（仅放跨域通用表，例如字典、序列号、审计表；业务表 mapper 仍归各业务模块）
  - `storage`：文件存储抽象（接口约束）

说明：当前阶段只产出文档与约束，不落代码。

## 4. 核心流程

### 4.1 写链路通用流程（事务内）

1. 参数校验与权限校验（业务模块完成）。
2. 数据库事务开启。
3. 写入/更新核心表（带状态机校验与唯一约束）。
4. 写入审计字段与操作记录（如需要）。
5. 提交事务。
6. 触发缓存失效/异步任务（在事务提交后执行，避免事务内调用外部依赖）。

### 4.2 异步一致性流程（最终一致）

适用：推荐刷新、审核执行、计数回写、消息投递等。

1. 业务侧先写本地事务（保存业务状态与任务引用）。
2. 异步执行器处理并落结果表/任务表。
3. 上游轮询查询或回调更新业务状态。

约束：禁止跨库分布式事务；以“任务表 + 幂等 + 状态机”实现最终一致。

## 5. 接口约束

持久化模块本身不对外提供 HTTP API。对业务模块提供的“接口约束”体现为以下统一约定：

- 所有写操作必须具备幂等策略，至少满足以下其一：
  - 业务唯一约束（unique key）
  - 幂等键（`Idempotency-Key`）映射到幂等表（后续可补）
  - 内容哈希（如审核 `content_hash`）
- 任何写入后会影响读热点的表，必须定义“写后失效”的缓存键集合（由各业务模块在自身文档中声明，并与缓存模块对齐）。
- 业务状态必须以“可枚举状态机”建模，不允许使用自由文本状态。

## 6. 数据约束

### 6.1 主键与通用字段规范

- 主键：统一使用 `bigint`（建议雪花/序列号，初期可用自增但需明确策略；同一项目保持一致）。
- 审计字段：
  - `created_at`（datetime，not null）
  - `updated_at`（datetime，not null）
  - 可选：`created_by`、`updated_by`（记录操作者 ID，管理端强烈建议开启）
- 软删除（可选）：`deleted`（tinyint）+ `deleted_at`；若使用软删，所有查询必须默认过滤。

### 6.2 核心表设计原则（按域）

用户域：

- `user_account`：鉴权相关字段单表隔离，避免资料表暴露敏感字段。
- `user_profile`：可变字段（昵称、头像、简介、等级）与账户分表。
- `user_message`（若采用）：消息投递与已读状态可独立表；写多读多需索引优化。

内容域：

- `novel`：小说主表存摘要类字段；大字段谨慎使用 `text`，避免频繁全表扫描。
- `novel_chapter`：章节元数据与正文大字段起步同表；当正文读写压力显著时，预留拆分为 `novel_chapter_content`。

互动域：

- `comment`：主评/回复用 `parent_id` 表示树；必须可分页查询。
- `reaction`（点赞/收藏）建议单独表：唯一约束 `uk_user_target_type_target_id`。
- `user_follow`：唯一约束 `uk_user_target_user`。

审核域：

- `audit_task`：快照与哈希必须可追溯；状态机明确；保留期与脱敏策略必须声明。

阅读域：

- `reading_progress`：唯一约束 `uk_user_novel`；写入需幂等（按时间戳或版本）。

### 6.3 状态字段与枚举约束

- `status` 字段统一使用 `tinyint` 或 `varchar(32)`（推荐 tinyint + 枚举映射，降低存储与索引成本）。
- 所有状态流转必须在业务侧做校验；数据库层可通过 `where status in (...)` 的更新条件实现并发保护。

## 7. 缓存、事务与一致性约束

### 7.1 事务约束

必须事务化的典型场景：

- 章节发布：章节状态变更 + 小说聚合字段（字数/最新章节）更新必须同事务。
- 上架/下架：状态变更 + 审计记录同事务。
- 关注/取关：关系表写入 + 计数更新（若计数落库）建议同事务；若计数走异步回写则只保证最终一致。
- 消息已读：状态变更必须幂等，避免重复更新导致写放大。

禁止在事务中做的事情：

- 调用外部服务（推荐、审核、对象存储）。
- 进行长耗时计算或批量全表扫描。

### 7.2 一致性约束

- 强一致：账户安全、内容发布状态、审核结论落库（任务终态）。
- 最终一致：点赞/收藏计数、榜单统计、推荐结果刷新、通知投递。

并发控制建议：

- 用唯一约束兜底（如 follow、reaction）。
- 用乐观锁或“状态机条件更新”避免重复发布/重复复审：
  - `update ... where id=? and status=?`
- 热点写（计数）优先在 Redis 聚合，异步落库；落库需幂等并可重放。

### 7.3 分库分表与读写分离预留

预留原则：

- 所有表必须具备清晰的分片键候选（优先 `user_id`、`novel_id`）。
- 避免跨域强 join；跨域关联采用“冗余字段 + 最终一致同步”。
- 高增长表（comment、message、audit_task、reading_progress）必须提前规划索引与归档策略。

## 8. 安全约束

- 严禁在日志中输出敏感字段：密码摘要、Token、审核内容快照全文、个人隐私字段。
- 对用户生成内容（简介、正文、评论）在入库前必须经过基础清洗与长度限制；具体审核由审核引擎负责。
- 文件存储必须做：
  - MIME/后缀双校验
  - 大小限制
  - 病毒/木马扫描（后续增强）
  - 权限控制与防盗链（建议签名 URL 或私有桶）

## 9. 异常、日志与审计约束

数据库异常处理：

- 唯一约束冲突：返回业务可理解错误码（归类 `3000-3999`），并在日志记录冲突键与 traceId。
- 外键/约束失败：归类参数或状态错误（`1000-1999` / `3000-3999`）。
- 连接/超时：归类系统错误（`5000-5999`），必须告警并记录慢 SQL 线索。

日志与审计最小字段集合：

- `traceId`、`operatorId`、`bizType`、`bizId`、`action`、`elapsedMs`、`dbRows`、`errorCode`（如失败）
- 对“状态变更/封禁/下架/复审”等敏感操作必须记录 `fromStatus/toStatus/reason`。

性能与慢 SQL 约束：

- 列表查询必须分页，禁止无条件全表扫描。
- 对热点查询必须有可用索引，并通过 `explain` 验证（落地阶段执行）。

## 10. 实现边界

- 本文档定义的是数据层规范与约束，不替代各业务模块的领域设计。
- 业务表的 Mapper/Repository 归各业务模块所有；持久化模块仅提供通用规范与少量跨域通用表（如字典/审计表）约定。
- 文件存储必须抽象为接口（如 `StorageService`），业务侧只依赖抽象，不依赖具体实现（本地/对象存储）。

## 11. 开发顺序

1. 明确 ID 策略（自增/雪花/序列号）与通用字段规范，并在所有核心表统一落地。
2. 按域落地核心表（user、novel、chapter、comment、follow/reaction、reading_progress、audit_task），并建立必要的唯一约束与索引。
3. 定义状态机枚举与“条件更新”并发保护原则，形成统一规范。
4. 定义文件存储目录/桶与访问策略，明确上传校验与签名 URL 方案（先文档约束，后实现）。
5. 补齐归档与保留策略：评论/消息/审核快照等高增长表的归档窗口与清理机制。
6. 在集成阶段对关键查询做索引验证与慢 SQL 基线建立（explain + 压测）。
