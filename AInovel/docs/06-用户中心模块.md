# 用户中心模块

## 1. 模块目标

- 提供统一的账号体系：注册、登录、登出、Token 刷新与登录态管理。
- 管理用户资料：昵称、头像、简介、等级、认证状态与偏好设置。
- 提供用户侧“消息中心”能力：系统通知、互动通知、审核通知的收件箱与已读状态。
- 对其他业务模块输出稳定的“用户身份与资料查询”能力，避免跨模块直接依赖账号表细节。

## 2. 职责边界

模块负责：

- 账号与认证：注册、登录、密码重置（后续）、Token 签发与撤销、登录态版本管理。
- 用户资料：资料维护、头像变更、用户状态（正常/禁用/封禁）管理。
- 消息中心：消息投递入口（供其他模块调用）、消息查询、已读/删除状态。
- 用户风控基础：登录失败计数、设备指纹/来源记录、异常登录审计事件。

模块不负责：

- 评论、点赞、关注等互动行为的业务写入（由社区互动模块负责）。
- 阅读进度的写入链路与高频读取（由阅读模块负责）。
- 推荐结果生成与推荐画像计算（由推荐引擎模块负责）。
- 审核规则与工单流转（由审核引擎模块负责）。

跨模块协作约束：

- 其他模块需要用户信息时，只能通过用户中心提供的查询接口或域服务获取（单体阶段允许同库读，但禁止直接写 `user_account/user_profile`）。
- 互动通知、审核通知由对应模块生成，但投递必须走用户中心“消息投递接口”，统一落库与已读状态。

## 3. 模块结构

建议包结构（以 `com.ainovel.user` 为根）：

- `controller`：`AuthController`、`UserProfileController`、`UserMessageController`。
- `service`：认证编排、资料编排、消息编排、登录审计与风控。
- `domain`：`UserAccount`、`UserProfile`、`UserStatus`、`UserRole`、`UserMessage`。
- `mapper`：MyBatis Mapper（`UserAccountMapper`、`UserProfileMapper`、`UserMessageMapper`）。
- `dto`：注册/登录/刷新/更新资料/消息已读请求。
- `vo`：`UserMeVO`、`UserProfileVO`、`UserMessageVO`。

模块对外最小能力面：

- 身份：`currentUserId`、`roles`、`status`。
- 资料：`nickname`、`avatarUrl`、`level`、`verifiedStatus`。

## 4. 核心流程

### 4.1 注册

1. 校验用户名/密码规则与唯一性。
2. 密码使用强哈希存储（例如 bcrypt/argon2，具体实现阶段确定）。
3. 写入 `user_account` 与 `user_profile`（昵称可默认与用户名一致）。
4. 返回注册结果（可选：直接签发登录态）。

约束：

- 注册接口必须限流，防止批量注册。
- 用户名不可枚举：重复用户名返回统一提示，不暴露存在性细节。

### 4.2 登录

1. 校验账号状态（禁用/封禁不可登录）。
2. 校验密码哈希。
3. 生成 Access Token（JWT）并返回过期时间。
4. 写入登录审计日志（不记录明文密码与敏感 Token）。

约束：

- 登录失败要做指数退避或验证码策略（后续实现），至少要记录失败次数与来源。
- 同账号多端登录策略需明确：默认允许多端，采用“登录态版本”支持强制下线。

### 4.3 登出与强制下线

1. 客户端调用登出接口。
2. 记录 Token 撤销信息（Redis 黑名单或登录态版本递增）。
3. 返回成功。

约束：

- 单点强制下线使用“登录态版本”优先，减少黑名单存储压力。
- 管理端强制下线必须审计记录原因与操作人。

### 4.4 Token 刷新

1. 客户端携带刷新凭证或旧 Token 调用刷新接口（具体机制实现阶段定）。
2. 校验凭证有效性、用户状态与登录态版本。
3. 签发新 Token 并返回。

约束：

- 刷新接口必须限流。
- 刷新凭证必须可撤销，且必须绑定用户与设备维度（设备维度后续实现）。

### 4.5 更新个人资料

1. 校验登录态与用户状态。
2. 校验字段长度、敏感字符与头像 URL 合法性。
3. 更新 `user_profile`。
4. 失效用户资料缓存。

约束：

- 昵称更新需做频率限制（例如 30 天内次数限制，后续实现）。
- 头像上传走文件存储模块，用户中心只保存 URL。

### 4.6 消息投递与已读

投递：

1. 其他模块调用“消息投递接口”，提供 `toUserId`、`type`、`title`、`content`、`bizRef`。
2. 用户中心落库 `user_message`，默认未读。
3. 可选：触发 WebSocket/推送（接入层能力）。

已读：

1. 用户批量或单条标记已读。
2. 更新 `read_at` 并返回结果。

约束：

- 投递接口必须鉴权为内部调用（管理端权限或服务间权限，单体阶段用角色与白名单控制）。
- 已读接口必须幂等：重复标记不改变语义。

## 5. 接口约束

通用约束：

- 统一前缀：`/api/v1/**`。
- 统一响应：`Result<T>`。
- DTO 入参必须 JSR 303 校验。
- 所有涉及写操作接口必须具备幂等策略或天然幂等语义。

建议接口：

- 注册：`POST /api/v1/auth/register`
  - 权限：匿名
  - 限流：必须（IP + 设备 + 用户名维度）

- 登录：`POST /api/v1/auth/login`
  - 权限：匿名
  - 限流：必须（IP + 用户名维度）

- 登出：`POST /api/v1/auth/logout`
  - 权限：登录用户

- 刷新 Token：`POST /api/v1/auth/refresh`
  - 权限：登录用户或刷新凭证
  - 限流：必须

- 获取当前用户：`GET /api/v1/users/me`
  - 权限：登录用户

- 更新资料：`PUT /api/v1/users/me/profile`
  - 权限：登录用户
  - 约束：昵称/简介长度限制，头像 URL 必须合法

- 查询消息列表：`GET /api/v1/users/me/messages?page=&size=&type=&readStatus=`
  - 权限：登录用户
  - 约束：必须分页

- 标记消息已读：`POST /api/v1/users/me/messages/read`
  - 权限：登录用户
  - 幂等：必须

- 投递消息（内部）：`POST /api/v1/internal/messages`
  - 权限：仅内部角色或白名单
  - 约束：必须带 `bizType/bizId` 便于追溯

## 6. 数据约束

### 6.1 核心表与字段约束

`user_account`（账号与鉴权）：

- `id`：主键。
- `username`：唯一，长度与字符集限制（禁止空白与控制字符）。
- `password_hash`：强哈希存储，禁止返回给客户端。
- `status`：`NORMAL`、`DISABLED`、`BANNED`。
- `role_mask` 或 `roles`：角色集合（RBAC）。
- `login_version`（建议新增）：登录态版本号，用于强制下线。

索引建议：

- `uk_username` 唯一索引。
- `idx_status_created_at` 普通索引。

`user_profile`（资料）：

- `user_id`：主键或唯一外键。
- `nickname`：长度限制，禁止敏感词（可接审核引擎做异步校验）。
- `avatar_url`：仅存 URL。
- `bio`：长度限制与 XSS 防护。
- `level`、`verified_status`：由业务规则更新。

`user_message`（消息中心，建议新增）：

- `id`：主键。
- `to_user_id`：收件人。
- `type`：`SYSTEM`、`INTERACTION`、`AUDIT`。
- `title`、`content`：长度限制，避免大字段滥用。
- `biz_type`、`biz_id`：业务引用（例如 `COMMENT/123`）。
- `read_at`：已读时间，空表示未读。
- `created_at`：创建时间。

索引建议：

- `idx_to_user_id_created_at`（消息列表分页）。
- `idx_to_user_id_read_at`（未读查询）。

### 6.2 状态与一致性约束

- 用户被封禁后：必须拒绝登录；已有登录态必须失效（通过 `login_version` 或黑名单）。
- 用户资料更新：必须写审计日志，并失效缓存。
- 消息投递：必须可追溯（`biz_type/biz_id/traceId`），禁止匿名投递。

## 7. 缓存、事务与一致性约束

缓存键建议：

- `user:profile:{userId}`：用户资料（TTL 分钟级，带随机抖动）。
- `auth:token_ver:{userId}`：登录态版本（或存于 DB，Redis 为加速）。
- `auth:token_blacklist:{jti}`：Token 黑名单（仅在需要时使用）。
- `user:msg_unread:{userId}`：未读计数（可选，最终一致）。

缓存策略：

- 资料更新走 Cache Aside：先更新 DB，再删除 `user:profile:{userId}`。
- 登出与封禁优先使用 `login_version`，黑名单仅覆盖短期 Token 撤销场景。

事务边界：

- 注册：`user_account` + `user_profile` 同事务。
- 标记已读：仅更新消息表，天然幂等，事务可选但建议批量操作同事务。

最终一致性：

- 未读计数允许最终一致，写入消息表后异步更新计数缓存（后续实现）。

## 8. 安全约束

- 密码安全：
  - 禁止明文存储与日志打印。
  - 哈希算法必须可配置，且包含盐与足够成本因子。
- Token 安全：
  - JWT 必须包含 `sub(userId)`、`roles`、`exp`、`jti`、`loginVersion`（或等价字段）。
  - 校验时必须比对 `loginVersion`，防止旧 Token 继续有效。
- 账号枚举防护：
  - 登录与注册的错误提示避免区分“用户不存在/密码错误”。
- 风控与限流：
  - 登录、注册、刷新 Token 必须限流。
  - 管理端对封禁/强制下线操作必须二次确认（交互后续实现）并审计。
- 输入安全：
  - 昵称、简介、消息内容必须做 XSS 基础处理与长度限制。
  - 头像 URL 必须白名单协议（`https` 优先）。

## 9. 异常、日志与审计约束

错误码分层（沿用全局约束）：

- `1000-1999`：参数错误（格式、长度、缺失字段）。
- `2000-2999`：认证授权错误（未登录、Token 失效、权限不足）。
- `3000-3999`：业务状态错误（账号禁用、封禁、操作不允许）。
- `5000-5999`：系统内部错误。

日志要求：

- 认证相关日志必须包含：`traceId`、`userId(如有)`、`username(脱敏)`、`clientIp`、`ua`、`result`、`elapsedMs`。
- 资料修改审计必须包含：`operatorId`、`targetUserId`、`changedFields`、`reason(如有)`。
- 消息投递审计必须包含：`toUserId`、`type`、`bizType/bizId`、`producer`、`traceId`。

敏感信息保护：

- 禁止在日志与接口返回中输出 `password_hash`、完整 Token、刷新凭证。

## 10. 实现边界

- 用户中心仅定义账号、资料、消息的模型与接口契约，不承诺具体前端形态或推送通道实现。
- 用户中心不直接实现“关注/收藏/阅读历史”的写入逻辑；如需要在用户中心页面展示，采用聚合查询或调用对应模块公开接口。
- 任何跨模块写入必须经过对应模块的用例层，不允许直接写对方表。

## 11. 开发顺序

1. 建立 `user_account/user_profile` 的约束与索引，补齐 `login_version` 字段设计。
2. 实现注册与登录流程的接口契约与校验规则，定义 JWT claims 规范。
3. 实现登出、强制下线与 Token 刷新契约，落地登录审计事件字段。
4. 实现资料查询与更新接口，补齐缓存键规范与失效策略。
5. 增加 `user_message` 表与消息中心接口（投递、分页查询、已读幂等）。
6. 补齐风控与限流策略的约束落地（与安全治理模块对齐），完善告警与审计字段。
