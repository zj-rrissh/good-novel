# 本地开发运行说明

适用范围：`dev` 本地联调，目标是最小可运行（MySQL + Redis + Spring Boot）。

本文档同时覆盖两类验证基线：

- 后端：`./mvnw spring-boot:run` 与 `./mvnw test`
- 前端：`npm run build` 与 `npm run test:contract`

如果你只想复核当前仓库的工程基线，直接看“验证基线”和“常见故障排
查”两节即可。

## 1. 前置条件

- JDK 17+ 的完整开发包（必须同时包含 `java` 和 `javac`，不要使用只有运行时的 JRE）
- Node.js 22 + npm
- Docker（推荐，用于快速起 MySQL/Redis）
- Bash 或 PowerShell

## 2. 环境变量约定

默认本地开发使用 `dev` profile。共享配置位于 `src/main/resources/application.properties`，本地开发数据源与缓存连接配置位于 `src/main/resources/application-dev.properties`。

以下变量已在配置文件中接入：

```bash
export AINOVEL_DB_URL='jdbc:mysql://127.0.0.1:3307/ainovel?useUnicode=true&characterEncoding=UTF-8&serverTimezone=Asia/Shanghai&useSSL=false&allowPublicKeyRetrieval=true'
export AINOVEL_DB_USERNAME='ainovel'
export AINOVEL_DB_PASSWORD='ainovel123'

export AINOVEL_REDIS_HOST='127.0.0.1'
export AINOVEL_REDIS_PORT='6380'
export AINOVEL_REDIS_DATABASE='0'
export AINOVEL_REDIS_PASSWORD=''
export AINOVEL_REDIS_ADDRESS='redis://127.0.0.1:6380'

# 认证相关敏感配置必须通过环境变量提供
export AINOVEL_JWT_SECRET='replace-with-a-random-secret'
```

说明：

- `AINOVEL_DB_PASSWORD`、`AINOVEL_REDIS_PASSWORD`、`AINOVEL_JWT_SECRET` 均为敏感信息，不要写入仓库。
- 首次初始化数据库时，可临时开启：
  `export AINOVEL_SQL_INIT_MODE='always'`
- 当前仓库默认 `docker compose` 暴露 MySQL `3307`、Redis `6380`，上面的默认值已经与其对齐。

## 3. 使用 Docker Compose 启动 MySQL 和 Redis

仓库根目录已提供 [compose.yaml](/mnt/d/myprogram/ai-novel/ainovel/compose.yaml)，默认会启动：

- MySQL 8.4
- Redis 7.2
- MySQL 首次建卷时自动执行 `src/main/resources/db/schema-dev.sql`

直接启动：

```bash
docker compose up -d mysql redis
```

查看状态：

```bash
docker compose ps
```

说明：

- 默认端口映射到宿主机 `3307/6380`，`application-dev.properties` 已与该映射对齐。
- 若需要改端口或密码，可在执行前导出 `AINOVEL_DB_PORT`、`AINOVEL_DB_PASSWORD`、`AINOVEL_REDIS_PORT`、`AINOVEL_REDIS_PASSWORD`、`AINOVEL_MYSQL_ROOT_PASSWORD`。
- `schema-dev.sql` 只会在 MySQL 数据卷首次初始化时自动执行；如果卷里已有旧数据，不会重复导入。

## 4. 备用方式：单独 Docker 命令

启动 MySQL 8：

```bash
docker run -d --name ainovel-mysql \
  -p 3306:3306 \
  -e MYSQL_ROOT_PASSWORD=root123 \
  -e MYSQL_DATABASE=ainovel \
  -e MYSQL_USER=ainovel \
  -e MYSQL_PASSWORD=ainovel123 \
  mysql:8.4 \
  --character-set-server=utf8mb4 \
  --collation-server=utf8mb4_0900_ai_ci
```

启动 Redis 7：

```bash
docker run -d --name ainovel-redis -p 6379:6379 redis:7.2
```

## 5. 初始化数据库

如果你使用上面的 `docker compose up -d mysql redis` 且 `mysql-data` 是首次创建的新卷，数据库会自动导入，一般不需要手动执行。

如果需要手动补导或重建，可用下面方式。

推荐方式 A（显式执行 SQL）：

```bash
mysql -h127.0.0.1 -P3307 -uainovel -painovel123 ainovel < src/main/resources/db/schema-dev.sql
```

如果本机没有安装 `mysql` CLI，可直接通过容器执行：

```bash
docker compose exec -T mysql mysql -uainovel -painovel123 ainovel < src/main/resources/db/schema-dev.sql
```

备选方式 B（由 Spring Boot 自动执行一次）：

1. 设置 `AINOVEL_SQL_INIT_MODE=always`
2. 启动应用一次
3. 建表完成后恢复 `AINOVEL_SQL_INIT_MODE=never`

如果要强制重新初始化 Docker 中的 MySQL 数据卷：

```bash
docker compose down -v
docker compose up -d mysql redis
```

## 6. 启动应用

```bash
./mvnw spring-boot:run
```

Windows:

```powershell
mvnw.cmd spring-boot:run
```

说明：

- 默认运行说明面向 `dev` 联调，`spring.profiles.default=dev`，数据源按本文档接入 MySQL。
- 自动化测试使用 `test` profile、H2 初始化脚本和测试替身 Bean，不依赖本地 MySQL/Redis。
- 若仅验证测试基线，可直接执行 `./mvnw test`。
- `mvnw` / `mvnw.cmd` 现在会在真正执行 Maven 前校验当前环境是否为完整 JDK；若缺少 `javac`，会直接终止并提示修复 `JAVA_HOME`。
- VS Code 根目录的 `AInovelApplication` 启动配置会先执行一次 `mvnw compile`，避免源码未编译时直接进入 `ClassNotFoundException`。

## 7. 验证基线

本节说明当前仓库的最小验证路径。你可以只验证后端，也可以把前后端基
线都跑一遍。

### 7.1 后端验证

在后端目录执行：

```bash
./mvnw test
```

说明：

- `mvnw` / `mvnw.cmd` 会先检查当前环境是否同时具备 `java` 和 `javac`
- 如果只找到 `java` 没找到 `javac`，wrapper 会直接失败，并提示你修复
  `JAVA_HOME` 或 PATH
- 这类失败属于环境前置条件不满足，不是后端测试本身失败

### 7.2 前端验证

在前端目录执行：

```bash
npm install
npm run build
npm run test:contract
```

说明：

- `npm run build` 与 `npm run test:contract` 现在都会先执行
  `npm run verify:native`
- `verify:native` 会检查当前平台所需的 `rollup` / `rolldown` 原生依赖
  是否真的安装到当前 `node_modules`
- 如果缺少当前平台所需依赖，命令会在构建或测试开始前直接失败，并给出
  恢复步骤

### 7.3 CI 验证

仓库根目录的 GitHub Actions 工作流现在会执行以下校验：

- 后端 `./mvnw test`
- 前端 `npm run build`
- 前端 `npm run test:contract`
- 接口契约检查 `node ai-context/scripts/check-api-contract.mjs`

CI 中前端统一使用干净安装 `npm ci`，避免复用跨平台污染的
`node_modules`。

## 8. 常见故障排查

本节按“后端 JDK 问题”和“前端原生依赖问题”两类整理。

### 8.1 `找不到或无法加载主类 com.ainovel.AInovelApplication`

这通常不是入口类丢了，而是编译阶段没有产出
`target/classes/com/ainovel/AInovelApplication.class`。

优先检查以下几项：

1. 确认当前 Java 环境是完整 JDK，而不是只有运行时的 JRE。
2. 在后端目录执行 `mvnw.cmd -v` 或 `./mvnw -v`，确认 Maven 使用的就
   是预期 JDK。
3. 执行 `javac -version`；如果命令不存在，说明当前终端或 IDE 没有接
   到完整 JDK。
4. 执行 `mvnw.cmd -DskipTests compile` 或 `./mvnw -DskipTests compile`，
   确认 `target/classes/com/ainovel/AInovelApplication.class` 已生成。
5. 如果你使用 VS Code，重新执行 “Java: Clean Java Language Server
   Workspace” 后再运行根目录下的 `AInovelApplication` 配置。

若 `JAVA_HOME` 未设置，可将其指向完整 JDK 17 安装目录，例如 Windows
下：

```powershell
$env:JAVA_HOME = 'C:\Program Files\Microsoft\jdk-17.0.9.8-hotspot'
$env:Path = "$env:JAVA_HOME\bin;$env:Path"
```

### 8.2 `npm run build` 或 `npm run test:contract` 提示缺少 native binding

如果你看到下面这类报错，通常不是 `vite` 或 `vitest` 本身坏了，而是当前
`node_modules` 里缺少当前平台所需的原生包：

- `Cannot find module @rolldown/binding-...`
- `Cannot find module @rollup/rollup-...`

最常见原因如下：

1. 你在另一个平台安装过依赖，例如先在 Windows 下装过，再在 WSL 里直
   接复用同一份 `node_modules`
2. 你当前的 `node_modules` 不是干净安装结果

推荐恢复步骤如下：

1. 删除当前前端目录下的 `node_modules`
2. 在当前平台重新执行 `npm install`
3. 再执行 `npm run build` 与 `npm run test:contract`
4. 在 CI 中始终使用 `npm ci`

如果你只想单独检查原生依赖是否齐全，可以执行：

```bash
npm run verify:native
```

### 8.3 快速健康检查

如果你已经启动后端应用，可执行：

```bash
curl http://127.0.0.1:8080/actuator/health
```

返回 `{"status":"UP"}` 说明应用已启动。

