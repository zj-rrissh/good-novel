# 社区互动模块

## 1. 模块目标

- 提供围绕小说与章节的互动能力：评论、回复、点赞、收藏（书架/收藏夹）、关注作者。
- 在高并发读与高频写（点赞/收藏）场景下保持可用性与可治理性：限流、防刷、审计、可追溯。
- 与审核引擎联动实现内容合规：对评论/回复文本做自动审核与人工复审闭环。

## 2. 职责边界

模块负责：

- 评论体系：对小说/章节发表评论、回复、删除（软删）、屏蔽、列表查询（分页、排序）。
- 互动行为：点赞/取消点赞、收藏/取消收藏、关注/取消关注。
- 计数与聚合：评论数、点赞数、收藏数、关注数等计数的读写与一致性策略。
- 互动风控：接口限流、幂等、反作弊信号记录（仅记录，不做复杂风控模型）。
- 审核协作：评论文本提交审核、应用审核结论（显示/隐藏/驳回原因摘要）。

模块不负责：

- 小说内容生产与发布（小说管理模块）。
- 章节正文读取与阅读进度（阅读模块）。
- 推荐策略与排序（推荐引擎模块，社区仅提供可消费的互动数据接口/事件）。
- 全站认证与 RBAC（安全治理模块提供鉴权能力；本模块只声明权限要求）。

禁止依赖：

- 禁止直接跨模块写入 `novel`、`novel_chapter`、`reading_progress` 等表。
- 禁止通过“直连对方内部 service/mapper”方式获取小说/章节可见性；必须通过公共查询或复用统一可见性校验组件（后续实现时落在 common 层）。

## 3. 模块结构

建议包结构（以 `com.ainovel.community` 为根）：

- `controller`
  - `CommentController`：评论/回复/列表
  - `ReactionController`：点赞/收藏/关注
  - `AdminCommunityController`：管理端屏蔽/解禁/审计查询（如需要）
- `service`
  - `CommentService`：评论写入、删除、审核协作
  - `ReactionService`：点赞/收藏/关注的幂等与计数
  - `CounterService`：计数读写、异步落库（若采用）
- `domain`
  - `Comment`, `CommentStatus`, `TargetType`（NOVEL/CHAPTER）
  - `ReactionType`（LIKE/FAVORITE/FOLLOW）
- `mapper`：MyBatis mapper
- `dto`：请求 DTO（创建评论、点赞、收藏、关注）
- `vo`：返回 VO（评论树、列表、计数摘要）

## 4. 核心流程

### 4.1 发表评论/回复

1. 鉴权：必须登录。
2. 可见性校验：目标小说/章节必须存在且可见（上架、未封禁、章节已发布等）。
3. 参数校验：长度、空白、敏感字符、频率约束。
4. 入库：写入 `comment` 表，初始状态 `PENDING_AUDIT` 或 `VISIBLE`（建议默认走审核，具体由配置开关控制）。
5. 审核协作：把评论文本提交审核引擎，异步更新审核结果。
6. 计数更新：评论数可先写 Redis 计数，再异步落库，或直接落库后异步回填缓存。

约束：

- 回复必须指向同一 `target_type/target_id` 的父评论，禁止跨目标串联。
- 不允许无限层级：建议仅允许两层（主评 + 回复）；更深层回复统一挂在主评下并记录 `reply_to_user_id`。
- 评论内容不得记录在访问日志中（仅记录摘要或长度）。

### 4.2 删除评论

1. 鉴权：作者本人可删除；管理端可屏蔽。
2. 软删：更新 `status=DELETED`，保留审计字段。
3. 计数：评论数按策略更新（建议：软删后计数异步修正，避免争用）。

约束：

- 删除为幂等：重复删除返回成功。
- 被审核驳回的评论允许用户删除或编辑重提（若支持编辑则必须重新审核）。

### 4.3 点赞/取消点赞

1. 鉴权：必须登录。
2. 幂等：同一用户对同一目标同一反应类型只能存在 0/1 条记录。
3. 写入反应表（或 Redis set）并更新计数。
4. 异步落库或直接落库（根据吞吐要求选择）。

约束：

- 点赞接口必须支持幂等键或天然幂等（重复点赞不重复计数）。
- 高频操作必须限流，防刷命中后写审计日志。

### 4.4 收藏/取消收藏

流程类似点赞，区别在于：

- 收藏用于书架/收藏夹入口，读取频率高且需支持分页查询。
- 需要支持“收藏时间排序”，因此最终必须落库（可先 Redis 缓冲再落库，但要能回放）。

### 4.5 关注/取消关注

1. 鉴权：必须登录。
2. 目标校验：不能关注自己；目标用户必须存在且未被封禁。
3. 幂等写入 `user_follow`，更新 follower/following 计数。

约束：

- 支持拉黑/屏蔽扩展点（当前只预留字段与错误码，不实现复杂关系网）。

## 5. 接口约束

通用约束：

- 统一前缀：`/api/v1/**`。
- 统一响应：`Result<T>`。
- 写接口必须鉴权，并启用双层限流（Nginx + 应用层）。
- 写接口必须幂等或提供 `Idempotency-Key` 请求头支持。

### 5.1 评论接口（建议）

- 发表评论/回复
  - `POST /api/v1/comments`
  - 参数：`targetType`、`targetId`、`content`、`parentId`（可选）、`replyToUserId`（可选）
  - 权限：登录用户
  - 幂等：建议支持 `Idempotency-Key`（避免弱网重试重复发评论）

- 删除评论
  - `DELETE /api/v1/comments/{commentId}`
  - 权限：评论作者或管理员
  - 幂等：天然幂等

- 查询评论列表
  - `GET /api/v1/{targetType}/{targetId}/comments?page=&size=&sort=hot|new`
  - 约束：必须分页；默认只返回 `VISIBLE` 评论；管理端可加 `status` 参数

### 5.2 互动接口（建议）

- 点赞/取消点赞
  - `POST /api/v1/reactions/like`
  - `POST /api/v1/reactions/unlike`
  - 参数：`targetType`、`targetId`
  - 幂等：重复调用不重复计数

- 收藏/取消收藏
  - `POST /api/v1/reactions/favorite`
  - `POST /api/v1/reactions/unfavorite`
  - 参数：`targetType=NOVEL`、`targetId`

- 关注/取消关注
  - `POST /api/v1/users/{targetUserId}/follow`
  - `POST /api/v1/users/{targetUserId}/unfollow`

### 5.3 计数查询（建议）

- 获取目标计数摘要
  - `GET /api/v1/counters?targetType=&targetId=`
  - 返回：`commentCount`、`likeCount`、`favoriteCount` 等
  - 约束：强制走缓存或聚合表，禁止每次实时 count(*) 回库

## 6. 数据约束

### 6.1 表：`comment`

核心字段建议：

- `id` bigint 主键
- `target_type` varchar(32) NOVEL/CHAPTER
- `target_id` bigint
- `user_id` bigint
- `parent_id` bigint 顶级为 0
- `reply_to_user_id` bigint 可选
- `content` text
- `status` tinyint：`PENDING_AUDIT`、`VISIBLE`、`HIDDEN`、`REJECTED`、`DELETED`
- `like_count` int 可选（建议由计数体系维护，不强依赖）
- `created_at` datetime
- `updated_at` datetime

索引与约束：

- `idx_target_created_at (target_type, target_id, created_at)` 支持新排序
- `idx_target_status (target_type, target_id, status)` 支持可见性过滤
- `idx_user_created_at (user_id, created_at)` 支持“我的评论”

### 6.2 表：`reaction`（点赞/收藏）

建议统一表承载点赞与收藏（也可拆表，当前建议统一便于治理）：

- `id` bigint
- `reaction_type` varchar(32)：LIKE/FAVORITE
- `target_type` varchar(32)：NOVEL/CHAPTER（收藏建议先只支持 NOVEL）
- `target_id` bigint
- `user_id` bigint
- `status` tinyint：ACTIVE/CANCELED
- `created_at` datetime
- `updated_at` datetime

唯一约束（保证幂等）：

- `uk_user_reaction (reaction_type, target_type, target_id, user_id)`

### 6.3 表：`user_follow`

- `id` bigint
- `user_id` bigint
- `target_user_id` bigint
- `status` tinyint：ACTIVE/CANCELED
- `created_at` datetime
- `updated_at` datetime

唯一约束：

- `uk_user_follow (user_id, target_user_id)`

### 6.4 状态与可见性约束

- 评论对外展示只允许 `VISIBLE`。
- `HIDDEN` 用于管理端屏蔽；`REJECTED` 用于审核驳回；`DELETED` 用于用户删除（软删）。
- 目标对象不可见时（小说下架、章节未发布），评论列表默认不可返回；管理端可按权限查看并操作。

## 7. 缓存、事务与一致性约束

### 7.1 计数体系

目标计数（comment/like/favorite/follow）禁止使用实时 `count(*)` 回库计算。

推荐方案（可分阶段落地）：

1. 起步版：写入 DB 后异步更新 Redis 计数缓存（读优先 Redis，失效后回源 DB 聚合表或预计算字段）。
2. 增强版：写入 Redis 计数（增量）并异步批量落库，使用“定时对账”修正漂移。

计数 Key 规范建议：

- `counter:comment:{targetType}:{targetId}`
- `counter:like:{targetType}:{targetId}`
- `counter:favorite:{targetType}:{targetId}`
- `counter:follow:following:{userId}`
- `counter:follow:follower:{userId}`

### 7.2 一致性规则

- 点赞/收藏/关注必须“用户维度幂等”，计数以“是否存在 ACTIVE 记录”为准。
- 异步落库场景必须可回放：所有增量事件需要落入可靠队列或落地日志（当前阶段可先用 DB 事务日志表，后续替换 MQ）。
- 允许短暂不一致：计数读允许秒级延迟，但写后读同用户维度应尽量一致（例如点赞后立即返回的计数可使用本地合成值）。

### 7.3 事务边界

- 写评论：评论入库与“本地审核任务引用写入”同事务；审核引擎调用不纳入分布式事务。
- 点赞/收藏/关注：如果采用 DB 直接写入，记录写入与计数落库（若同库）可在同事务；若计数走 Redis，则以最终一致性为主。

## 8. 安全约束

- 接口鉴权：所有写接口必须登录态；管理端接口必须 RBAC。
- 反作弊与防刷：
  - 评论发布、点赞、关注必须限流（IP + userId + endpoint）。
  - 发现异常（短时间大量点赞/评论）必须记录审计事件并可封禁（封禁规则由安全治理模块落地）。
- 内容安全：
  - 评论内容必须做 XSS 基础处理与长度限制（例如 1-500 字，可配置）。
  - 禁止在错误信息中回显原始违规内容。

## 9. 异常、日志与审计约束

日志字段建议（所有写接口）：

- `traceId`、`requestId`
- `userId`、`ip`、`ua`（摘要）
- `action`（COMMENT_CREATE/COMMENT_DELETE/LIKE/FAVORITE/FOLLOW）
- `targetType`、`targetId`
- `result`（SUCCESS/FAIL）
- `latencyMs`

审计事件（必须落地）：

- 评论发布/删除/屏蔽/解禁
- 点赞/收藏/关注的限流拦截与风控命中
- 审核回写异常（状态不匹配、重复回写、回写失败）

错误码约束（与全局约定一致）：

- `1000-1999`：参数错误（内容过长、目标类型非法）
- `2000-2999`：鉴权/权限错误
- `3000-3999`：业务状态错误（目标不可见、重复关注、评论已删除等）
- `5000-5999`：系统错误

## 10. 实现边界

- 本模块只定义互动域的数据模型、接口与约束，以及计数体系与审核联动边界。
- 不实现推荐排序算法；仅提供可消费的计数/列表接口与审计数据。
- 不实现复杂风控模型；仅提供限流、幂等、审计与扩展点。

## 11. 开发顺序

1. 定义数据表与唯一约束：`comment`、`reaction`、`user_follow`，并明确评论状态枚举。
2. 实现评论创建/删除/列表接口（带分页、可见性校验、基础限流）。
3. 接入审核引擎：评论发布进入 `PENDING_AUDIT`，回写后变更为 `VISIBLE/REJECTED/HIDDEN`。
4. 实现点赞/收藏/关注幂等写入与计数更新（先 DB 直写 + Redis 缓存读，后续再做异步批量）。
5. 增加计数摘要接口与缓存策略，补齐对账/漂移修正的定时任务约束（基础设施模块）。
6. 完善审计日志与管理端屏蔽/解禁接口（如需要运营治理能力）。
