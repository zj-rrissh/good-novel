# 小说管理模块

## 1. 模块目标

- 为作者与运营提供小说全生命周期管理能力：创建、编辑、章节管理、发布与上下架。
- 为阅读模块提供“可发布、可阅读”的内容主数据与目录数据，保证状态一致性与可追溯。
- 与审核引擎协作实现内容合规：简介、章节正文、封面等内容在发布前完成审核闭环。

## 2. 职责边界

模块负责：

- 小说（Novel）基础信息：标题、简介、封面、分类、标签、作者归属、状态流转。
- 章节（Chapter）管理：新增、编辑、保存草稿、发布、目录排序与章节元数据维护。
- 发布流程编排：提交审核、审核结果回写、发布/撤回、上下架。
- 管理端基础能力：列表查询、条件筛选、状态变更审计记录。

模块不负责：

- 章节正文的高频读取与阅读进度（由阅读模块负责读取链路与缓存策略）。
- 评论、点赞、收藏等互动（由社区互动模块负责）。
- 推荐结果生成（由推荐引擎模块负责）。
- 审核规则、人工复审（由审核引擎模块负责）；本模块仅负责“提交审核”和“应用审核结论”。

禁止依赖：

- 不允许跨模块直接写入 `reading_progress`、`comment` 等非本域表。
- 不允许直接调用推荐、社区模块内部服务，只能通过其公开 API/事件协作（后续实现时再定）。

## 3. 模块结构

建议包结构（以 `com.ainovel.novel` 为根）：

- `controller`：作者端、管理端接口（建议分 `AuthorNovelController`、`AdminNovelController`）。
- `service`：用例层，编排事务、审核协作、缓存失效。
- `domain`：领域对象与状态机（`Novel`, `Chapter`, `NovelStatus`, `ChapterStatus`）。
- `mapper`：MyBatis Mapper（`NovelMapper`, `ChapterMapper`, `NovelTagMapper`）。
- `dto`：请求 DTO（创建/更新/提交审核/发布）。
- `vo`：返回 VO（列表摘要、详情、章节目录）。
- `converter`：DTO/DO/VO 转换。

## 4. 核心流程

### 4.1 创建小说（草稿）

1. 作者提交标题、简介、分类、标签、封面等基础信息。
2. 写入 `novel` 表，状态为 `DRAFT`。
3. 返回小说 ID。

约束：

- 标题、简介做长度与敏感字符校验（JSR 303）。
- 创建后允许不含章节的空草稿存在。

### 4.2 编辑小说基础信息

1. 校验作者身份与归属（`author_id` 必须一致）。
2. 校验当前状态是否允许编辑（建议：`DRAFT`、`REJECTED` 可编辑；`PUBLISHED` 仅允许部分字段编辑，触发重新审核策略见 7.3）。
3. 更新 `novel` 表字段并记录审计日志。
4. 失效相关缓存（详见 7.1）。

### 4.3 新增/编辑章节（草稿）

1. 作者新增章节（`chapter_no`、标题、正文）。
2. 章节写入 `novel_chapter`，状态 `DRAFT`。
3. 编辑章节仅允许 `DRAFT` 或 `REJECTED`。

约束：

- `uk_novel_id_chapter_no` 保证章节序号唯一。
- 正文为大字段，不进入 L1 本地缓存（正文读取缓存由阅读模块负责，但写入仍在本模块）。

### 4.4 提交审核

提交对象：

- 小说简介（`novel.intro`）、封面（可选）、待发布章节正文。

流程：

1. 组装审核内容快照（文本、元数据、提交人）。
2. 调用审核引擎接口（同步返回“受理成功”即可，审核执行异步）。
3. 将小说/章节状态置为 `PENDING_AUDIT`（或章节 `PENDING_AUDIT`），记录 `audit_task_id`（如有）。

约束：

- 提交审核必须幂等：同一 `biz_type + biz_id + content_hash` 在审核未终结前重复提交应返回同一受理结果。
- 审核快照必须可追溯：审核引擎存储快照，本模块只保存引用 ID 与最后一次提交时间。

### 4.5 审核结果回写与发布

审核引擎回调或轮询更新：

- 通过：章节状态 `PUBLISHED`，小说若满足“至少 1 章已发布且小说审核通过”可置 `PUBLISHED`。
- 驳回：章节/小说状态 `REJECTED`，保存驳回原因摘要（不保存敏感规则细节）。
- 人工复审：状态保持 `PENDING_AUDIT` 或 `MANUAL_REVIEW`（由审核模块定义）。

发布约束：

- 发布动作必须事务化：章节发布与小说 `latest_chapter_id/word_count` 等聚合字段更新同事务。
- 发布后立即触发缓存失效与预热（由缓存模块约束，见 7.1）。

### 4.6 上架/下架

- 上架：仅 `PUBLISHED` 且未被封禁的小说可上架，状态 `ON_SHELF`。
- 下架：运营或作者（视权限）可下架，状态 `OFF_SHELF`。

约束：

- 下架后的内容对阅读模块应不可见（阅读模块查询必须过滤状态）。
- 状态变化必须写审计日志，并可追溯操作人、来源、原因。

## 5. 接口约束

通用约束：

- 统一前缀：`/api/v1/**`。
- 统一响应：`Result<T>`。
- 需要鉴权：除公开查询（如小说目录摘要）外，作者端与管理端接口均要求 JWT 登录态。
- 幂等性：创建/提交审核/发布等写接口必须支持幂等 key（建议：`Idempotency-Key` 请求头）。

### 5.1 作者端接口（建议）

- 创建小说
  - `POST /api/v1/novels`
  - 幂等：支持 `Idempotency-Key`
  - 权限：`ROLE_AUTHOR`
  - 约束：标题必填；分类必填；标签可选；封面可选

- 更新小说
  - `PUT /api/v1/novels/{novelId}`
  - 幂等：天然幂等
  - 权限：作者归属校验 + `ROLE_AUTHOR`

- 新增章节
  - `POST /api/v1/novels/{novelId}/chapters`
  - 幂等：支持 `Idempotency-Key`
  - 权限：作者归属校验 + `ROLE_AUTHOR`
  - 约束：`chapter_no` 必须连续或允许跳号需明确（建议允许跳号但发布时校验）

- 更新章节
  - `PUT /api/v1/chapters/{chapterId}`
  - 权限：作者归属校验 + `ROLE_AUTHOR`
  - 约束：仅 `DRAFT/REJECTED` 可编辑

- 提交审核
  - `POST /api/v1/novels/{novelId}/submit-audit`
  - 权限：作者归属校验 + `ROLE_AUTHOR`
  - 幂等：同内容重复提交返回同一受理结果

- 上架/下架
  - `POST /api/v1/novels/{novelId}/on-shelf`
  - `POST /api/v1/novels/{novelId}/off-shelf`
  - 权限：作者归属校验 + `ROLE_AUTHOR`（运营权限另见管理端）

### 5.2 管理端接口（建议）

- 条件查询小说列表
  - `GET /api/v1/admin/novels?status=&authorId=&categoryId=&keyword=&page=&size=`
  - 权限：`ROLE_ADMIN` 或运营角色
  - 约束：必须分页，禁止全量导出接口直接暴露

- 强制下架/封禁
  - `POST /api/v1/admin/novels/{novelId}/ban`
  - 权限：`ROLE_ADMIN`
  - 约束：需要原因字段，必须审计

## 6. 数据约束（数据模型约束）

核心实体：

- `novel`
  - `author_id` 不可变（创建后禁止修改）。
  - `status`：建议枚举 `DRAFT`、`PENDING_AUDIT`、`REJECTED`、`PUBLISHED`、`ON_SHELF`、`OFF_SHELF`、`BANNED`。
  - `word_count`、`latest_chapter_id` 作为聚合字段，仅由发布流程更新。
  - 索引：`idx_author_status`、`idx_category_status`、`idx_updated_at`。

- `novel_chapter`
  - `novel_id` 归属不可变。
  - `status`：`DRAFT`、`PENDING_AUDIT`、`REJECTED`、`PUBLISHED`。
  - 唯一约束：`uk_novel_id_chapter_no`。
  - 正文 `content` 为大字段，数据库层建议单表存储起步，后续可拆“章节内容表”。

标签与分类：

- 分类、标签为字典数据，ID 必须在字典表存在（字典表可由基础设施或运营模块提供）。
- `novel` 与 `tag` 采用关联表（如 `novel_tag`），更新时使用“先删后插”并保持事务一致性。

状态流转约束：

- `DRAFT -> PENDING_AUDIT -> (PUBLISHED | REJECTED)`
- `PUBLISHED -> ON_SHELF -> OFF_SHELF`（或 `ON_SHELF <-> OFF_SHELF`）
- 任何状态 `-> BANNED` 仅管理端触发
- `BANNED` 禁止回退，需走解禁流程（后续定义）

## 7. 缓存、事务与一致性约束

### 7.1 缓存策略（与缓存模块协作）

本模块负责“写后失效”，不负责阅读态正文缓存命中（读取由阅读模块负责）。

缓存键建议（写后必须失效）：

- `novel:detail:{novelId}`：小说详情
- `novel:chapters:{novelId}`：章节目录（不含正文）
- `recommend:home:{userId}`：当小说上架/下架对推荐可能有影响时，仅发送“推荐失效事件”，不直接操作推荐缓存

失效时机：

- 更新小说基础信息：失效 `novel:detail:{novelId}`。
- 新增/编辑章节：失效 `novel:chapters:{novelId}`；发布章节后由阅读模块负责失效 `chapter:content:{chapterId}`（或由事件触发）。
- 上架/下架：失效 `novel:detail:{novelId}`，并触发榜单/推荐异步刷新。

### 7.2 事务边界

必须使用 `@Transactional` 的场景：

- 创建小说并初始化关联数据（如标签关系）。
- 新增章节并更新小说统计字段（若同事务需要更新）。
- 发布章节：章节状态变更 + 小说聚合字段更新 + 审计记录写入。
- 上架/下架：状态变更 + 审计记录写入。

禁止跨事务协作：

- 审核引擎调用与本地 DB 更新不做分布式事务。以“本地状态 + 审核任务 ID”实现最终一致性。

### 7.3 一致性与重试

审核回写一致性：

- 审核回调必须幂等：同一 `audit_task_id` 重复回调不会造成重复发布或状态错乱。
- 状态机校验：仅允许从 `PENDING_AUDIT` 应用审核结论，其他状态收到回调应记录告警并丢弃或进入人工核对队列。

发布并发控制：

- 同一小说的章节发布需串行化（建议按 `novelId` 加锁，锁实现由缓存模块提供）。

## 8. 安全约束

- 作者端接口必须校验“作者归属”：`novel.author_id == currentUserId`。
- 管理端接口必须 RBAC 控制，敏感操作（封禁、强制下架）必须二次确认（后续交互实现）。
- 输入安全：
  - 简介、章节正文、标题必须做 XSS 基础处理与长度限制。
  - 禁止在接口返回中泄露审核规则命中细节，仅返回可理解的原因摘要。
- 防刷：
  - 创建/提交审核接口应受限流保护（接入层与应用层双层限流）。

## 9. 异常、日志与审计约束

日志要求：

- 所有状态变更记录业务审计日志：`operatorId`、`operatorRole`、`novelId`、`chapterId`（可选）、`fromStatus`、`toStatus`、`reason`、`traceId`。
- 写接口记录耗时与关键参数摘要（不记录全文正文）。
- 审核回写失败或状态不匹配必须告警（基础设施模块接管告警）。

异常处理：

- 参数错误：返回 `1000-1999` 段错误码。
- 权限不足：`2000-2999`。
- 状态不允许操作：`3000-3999`（例如“章节已发布不可编辑”）。
- 系统错误：`5000-5999`，并记录堆栈与 traceId。

## 10. 实现边界

- 本模块仅定义：
  - DB 模型与约束（表结构、索引、状态机）。
  - API 契约与权限/幂等/限流要求。
  - 与审核引擎的交互边界（提交审核、接收结论）。
  - 写后缓存失效原则。
- 不在本模块内实现：
  - 推荐算法与推荐数据生成。
  - 阅读侧缓存命中与章节正文的高频读取优化（阅读模块负责）。
  - 全站统一限流与风控的具体算法实现（安全治理模块负责）。

## 11. 开发顺序

1. 建立 `novel`、`novel_chapter` 基础表与状态枚举，完成作者归属校验的基础工具能力（依赖安全治理模块的登录态）。
2. 实现创建/编辑小说与章节草稿的用例层编排，补齐审计日志落地（依赖基础设施模块日志约束）。
3. 增加提交审核接口与审核任务引用字段，打通与审核引擎的“受理”交互。
4. 实现审核结果回写与发布流程（含幂等、状态机校验、事务边界）。
5. 增加上架/下架与管理端列表筛选接口，补齐缓存失效与热点预热触发点。
6. 补充压测关注点：发布并发、章节写入大字段、目录查询分页与索引效果验证。
