# 智能小说平台模块方案设计

## 1. 文档说明

- 目标：基于架构图输出一份可直接指导后续开发的模块设计方案。
- 适用范围：单体架构阶段的智能小说平台，支撑 PC Web 与移动 H5 访问。
- 当前项目基线：仓库目前是轻量 Spring Boot 骨架，本文档描述的是目标态设计，后续开发按此逐步补齐模块与依赖。

## 2. 技术方案概述

平台采用“`Nginx 接入层 + Spring Boot 单体应用 + MySQL + Redis + Caffeine + 文件存储`”的分层架构。整体以单体优先、模块内聚、缓存前置、异步削峰为核心原则，先满足中等规模用户并发与快速迭代，后续再按业务增长拆分推荐、审核、社区等重负载模块。

系统主要由以下几层组成：

1. 用户接入层：PC Web、移动 H5，通过 HTTP/HTTPS 接入，阅读态和通知态可扩展 WebSocket。
2. 边缘接入与安全治理层：由 Nginx、Spring Security、全局过滤器、限流模块组成，负责静态资源分发、认证授权、流量控制与安全防护。
3. 核心业务层：涵盖小说管理、阅读、社区互动、用户中心四类核心业务。
4. 智能服务层：包括章节推荐引擎与内容审核引擎，负责提升内容分发效率和平台合规能力。
5. 缓存层：采用 Caffeine + Redis + 布隆过滤器的三级缓存体系，缓解热点读写压力。
6. 持久化层：以 MySQL 8.0 为主存储，文件资源独立存放，支撑业务数据与静态资源管理。
7. 基础设施层：包含日志、监控、定时任务、异常处理等运行保障能力。

## 3. 技术选型

| 类别 | 选择 | 理由 |
|------|------|------|
| 语言 | Java 17 | 与当前项目保持一致，兼顾稳定性与生态成熟度 |
| 核心框架 | Spring Boot 4.x | 适合单体架构快速搭建，便于整合 Web、缓存、安全、任务调度 |
| ORM/数据访问 | MyBatis | 贴合架构图，适合复杂 SQL、分页查询与性能可控的读写场景 |
| 数据库 | MySQL 8.0 | 适合用户、小说、章节、互动等强一致业务数据 |
| 分布式缓存 | Redis | 支撑 Session、Token、推荐结果、热点章节缓存 |
| 本地缓存 | Caffeine | 适合 JVM 级热点数据缓存，降低 Redis 往返开销 |
| 安全框架 | Spring Security + JWT + RBAC | 统一认证授权链路，支持前后端分离 |
| 参数校验 | Bean Validation / JSR 303 | 统一 DTO 校验，减少接口脏数据 |
| 反向代理 | Nginx | 处理 HTTPS、静态资源、反向代理、限流与 CDN 协同 |
| 富文本/文件 | 本地文件系统或对象存储兼容层 | 支撑封面、头像、章节插图、富文本附件 |
| 日志监控 | Logback + ELK + Actuator | 满足日志检索、接口追踪和健康检查 |
| 异步任务 | Spring `@Async` + Spring Task | 满足审核、推荐刷新、消息通知等异步处理需求 |

## 4. 整体架构设计

### 4.1 分层结构

```text
客户端层
├── PC Web（HTTP/HTTPS）
└── 移动 H5（HTTP/HTTPS + WebSocket）

接入层
└── Nginx / CDN / 静态资源分发 / SSL 终止 / 限流

应用层
├── 安全治理模块
├── 核心业务模块
└── 智能服务模块

缓存层
├── L1 Caffeine 本地缓存
├── L2 Redis 分布式缓存
└── L3 布隆过滤器与穿透防护

持久化层
├── MySQL 8.0
└── 文件存储

基础设施层
├── 日志与链路追踪
├── 性能监控
├── 定时任务
└── 全局异常与统一响应
```

### 4.2 架构原则

1. 单体优先：当前阶段以单体架构承载全部核心能力，降低部署与联调复杂度。
2. 模块解耦：通过包结构、服务边界、领域对象进行逻辑解耦，为未来拆分微服务预留边界。
3. 缓存优先：对高频读取的小说详情、章节内容、推荐列表、用户会话进行分级缓存。
4. 读写分治：写操作走数据库事务，读操作优先命中缓存，避免核心表被高频查询拖垮。
5. 异步削峰：推荐计算、敏感审核、消息通知、统计聚合等非强实时任务改为异步执行。

## 5. 模块划分与方案设计

## 5.1 用户接入层

### 5.1.1 PC Web 模块

- 职责：提供创作者后台、运营后台、通用用户前台页面访问入口。
- 通信方式：HTTP/HTTPS。
- 适用场景：小说创作、章节编辑、用户中心、运营管理。
- 关键要求：
  - 页面首屏尽量静态化或做资源预加载。
  - 管理后台与前台接口按 `/api/admin/**`、`/api/app/**` 分域管理。
  - 登录态通过 JWT 或短期 Access Token + 刷新机制维持。

### 5.1.2 移动 H5 模块

- 职责：承载移动端阅读、追更、评论、互动、消息提醒。
- 通信方式：HTTP/HTTPS，消息场景可扩展 WebSocket。
- 关键要求：
  - 阅读页接口必须轻量，支持分页/分章按需加载。
  - 重点优化弱网环境下的章节加载、历史进度恢复与缓存命中。
  - 通知类功能可通过 WebSocket 或轮询平滑接入。

### 5.1.3 接口约束

- 所有对外接口统一使用 `/api/v1/**` 版本前缀。
- 响应统一通过 `Result<T>` 封装。
- 阅读类接口必须支持幂等和限流，避免章节接口被恶意刷取。

## 5.2 边缘接入与流量治理模块

### 5.2.1 Nginx 反向代理模块

- 职责：
  - SSL 终止，统一入口域名。
  - 静态资源与动态接口分流。
  - 反向代理 Spring Boot 服务。
  - 配置简单限流、黑白名单与恶意请求拦截。
- 关键配置建议：
  - `/static/`、`/assets/` 走静态资源目录或 CDN。
  - `/api/` 反向代理到应用服务。
  - `/ws/` 升级为 WebSocket 连接。
  - 对登录、评论、发布等接口设置 `limit_req`。

### 5.2.2 CDN 与静态资源加速

- 职责：缓存封面图、头像、站点静态资源、常见脚本文件。
- 方案：
  - 强缓存资源带 hash。
  - 封面、头像、章节插图使用独立资源域名。
  - 静态资源与业务接口彻底隔离，降低主应用带宽压力。

### 5.2.3 接入层限流策略

- 对象维度：IP、用户 ID、设备 ID、接口路径。
- 核心场景：
  - 登录接口防爆破。
  - 章节详情接口防爬取。
  - 评论/点赞接口防刷量。
- 策略建议：
  - Nginx 做第一层粗粒度限流。
  - 应用层做第二层精细化限流和风控记录。

## 5.3 安全治理模块

### 5.3.1 认证授权模块

- 技术方案：Spring Security + JWT + RBAC。
- 职责：
  - 用户登录认证。
  - 管理端与普通用户端权限隔离。
  - 基于角色控制创作、审核、运营等后台能力。
- 核心设计：
  - 登录成功后签发 Access Token。
  - Token 中包含用户 ID、角色、过期时间、签发版本。
  - Redis 可选存储黑名单 Token、刷新凭证、登录态版本。

### 5.3.2 全局过滤器模块

- 职责：
  - 请求日志记录。
  - TraceId 注入。
  - XSS 字符转义。
  - 基础 SQL 注入风险拦截。
  - 用户端设备/来源识别。
- 关键实现：
  - `OncePerRequestFilter` 记录请求耗时和上下文。
  - 统一提取 `requestId`，用于问题排查与日志检索。

### 5.3.3 接口限流模块

- 技术方案：令牌桶/漏桶算法 + Redis 计数。
- 处理流程：
  1. 进入 Controller 前读取用户/IP/接口配置。
  2. 判断当前配额是否超限。
  3. 超限则直接返回友好错误码。
  4. 命中风控阈值则记录异常行为日志。

## 5.4 核心业务模块

## 5.4.1 小说管理模块

- 职责：
  - 小说创建、编辑、删除。
  - 分类、标签、封面、简介管理。
  - 上架、下架、草稿、待审核等状态流转。
  - 章节目录维护与发布。
- 子模块：
  - 小说基础信息管理
  - 标签与分类管理
  - 章节管理
  - 发布审核状态管理
- 核心流程：
  1. 作者创建小说草稿。
  2. 维护分类、标签、简介与封面。
  3. 新增/编辑章节。
  4. 提交审核或直接发布。
  5. 审核通过后进入可阅读状态。
- 关键接口：
  - `POST /api/v1/novels`
  - `PUT /api/v1/novels/{novelId}`
  - `POST /api/v1/novels/{novelId}/chapters`
  - `PUT /api/v1/chapters/{chapterId}`
  - `POST /api/v1/novels/{novelId}/submit-audit`

## 5.4.2 阅读模块

- 职责：
  - 小说详情展示。
  - 章节列表查询。
  - 章节正文加载。
  - 阅读进度、书签、历史记录恢复。
- 关键设计：
  - 章节正文按章缓存，避免频繁回源数据库。
  - 阅读记录按“用户 + 小说”维度存储。
  - 支持分页章节目录查询，防止大部头小说一次性加载过重。
- 核心接口：
  - `GET /api/v1/novels/{novelId}`
  - `GET /api/v1/novels/{novelId}/chapters`
  - `GET /api/v1/chapters/{chapterId}/content`
  - `POST /api/v1/reading-progress`

## 5.4.3 社区互动模块

- 职责：
  - 评论、回复、点赞、收藏、关注。
  - Feed 流或动态聚合展示。
  - 互动消息触发。
- 关键设计：
  - 评论采用主评 + 回复结构。
  - 点赞/收藏可使用 Redis 做计数缓冲，再异步落库。
  - 高风险文本需进入内容审核引擎。
- 核心接口：
  - `POST /api/v1/comments`
  - `GET /api/v1/novels/{novelId}/comments`
  - `POST /api/v1/comments/{commentId}/like`
  - `POST /api/v1/users/{targetUserId}/follow`

## 5.4.4 用户中心模块

- 职责：
  - 注册、登录、登出。
  - 个人资料、头像、等级、认证信息管理。
  - 消息通知、收藏、关注列表、阅读历史管理。
- 关键设计：
  - 用户资料与鉴权信息分表设计，降低敏感字段暴露范围。
  - 消息中心支持系统通知、互动通知、审核通知三类消息。
  - 阅读历史与收藏列表优先读缓存，异步刷新统计数据。
- 核心接口：
  - `POST /api/v1/auth/register`
  - `POST /api/v1/auth/login`
  - `GET /api/v1/users/me`
  - `PUT /api/v1/users/me/profile`
  - `GET /api/v1/users/me/messages`

## 5.5 智能服务模块

## 5.5.1 章节推荐引擎

- 职责：
  - 基于用户画像、阅读历史、收藏行为生成推荐结果。
  - 支持首页推荐、同类推荐、追更推荐、个性化书单。
- 推荐策略：
  - 初期：分类标签 + 热度分 + 最近更新的规则推荐。
  - 中期：协同过滤 + 用户画像打分。
  - 高阶：引入内容向量和召回/排序分层。
- 处理方式：
  - 在线查询优先读取 Redis 推荐结果。
  - 推荐结果异步计算，定时刷新。
  - 新用户走冷启动策略，按热度与分类偏好推荐。
- 核心接口：
  - `GET /api/v1/recommend/home`
  - `GET /api/v1/recommend/novels/{novelId}/related`

## 5.5.2 敏感审核引擎

- 职责：
  - 对小说简介、章节正文、评论内容执行敏感词过滤与审核。
  - 将高风险内容转人工复审。
- 审核流程：
  1. 文本提交后进入审核队列。
  2. 规则引擎执行敏感词、违规模式检测。
  3. 命中轻度风险则自动拦截并返回修改建议。
  4. 命中重度风险则进入人工复审工单。
- 关键设计：
  - 审核结果必须可追溯，保留命中规则与时间戳。
  - 审核过程异步执行，避免阻塞主业务写入链路。
- 核心接口：
  - `POST /api/v1/audit/text`
  - `GET /api/v1/admin/audit/tasks`
  - `POST /api/v1/admin/audit/tasks/{taskId}/review`

## 5.6 缓存模块设计

### 5.6.1 L1 本地缓存

- 技术：Caffeine。
- 适用数据：
  - 分类列表
  - 标签字典
  - 热门小说摘要
  - 小说基础信息
- 设计要求：
  - TTL 控制在秒级到分钟级。
  - 只缓存热点轻量数据，不缓存大文本正文。

### 5.6.2 L2 分布式缓存

- 技术：Redis。
- 适用数据：
  - Token / Session
  - 小说详情
  - 章节目录
  - 推荐列表
  - 评论计数、点赞计数
  - 阅读历史临时态
- Key 设计建议：
  - `auth:token:{userId}`
  - `novel:detail:{novelId}`
  - `novel:chapters:{novelId}`
  - `chapter:content:{chapterId}`
  - `recommend:home:{userId}`

### 5.6.3 L3 穿透防护层

- 技术：布隆过滤器 + 空值缓存。
- 使用场景：
  - 非法小说 ID、章节 ID 被恶意扫描。
  - 热门接口频繁查询不存在资源。
- 设计要求：
  - 小说、章节等核心主键进入布隆过滤器。
  - 查无数据时短暂缓存空值，防止连续穿透。

### 5.6.4 缓存更新策略

- 模式：`Cache Aside Pattern`。
- 写流程：
  1. 先更新数据库。
  2. 再删除缓存或更新缓存。
  3. 重要热点数据可做异步预热。
- 风险控制：
  - 使用随机过期时间避免雪崩。
  - 对热点 Key 加互斥锁或逻辑过期防击穿。

  - `/avatars/`
  - `/richtext/`
- 设计要求：
  - 数据库存储文件访问 URL，不存大文件二进制。
  - 上传后做文件类型校验与大小限制。
  - 后续可平滑切换到对象存储。

## 5.8 基础设施模块

### 5.8.1 日志系统

- 方案：Logback 输出结构化日志，接入 ELK。
- 内容范围：
  - 应用日志
  - 接口访问日志
  - 异常堆栈日志
  - 安全审计日志
- 关键要求：
  - 每次请求记录 `traceId`。
  - 高风险操作记录操作人、目标对象、结果与耗时。

### 5.8.2 性能监控

- 方案：Spring Boot Actuator。
- 暴露指标：
  - JVM 内存
  - 线程池状态
  - Redis/MySQL 健康状态
  - HTTP 请求耗时
  - 缓存命中率

### 5.8.3 定时任务模块

- 方案：Spring Task。
- 任务类型：
  - 推荐结果刷新
  - 热门小说榜单重算
  - 审核积压检查
  - 失效缓存预热
  - 历史统计归档

### 5.8.4 异常处理模块

- 方案：`@ControllerAdvice` + 统一错误码体系。
- 目标：
  - 对参数错误、权限错误、业务异常、系统异常做分层返回。
  - 降低前端与调用方解析复杂度。

## 5.9 横切关注点设计

| 横切能力 | 方案 | 说明 |
|----------|------|------|
| 统一响应封装 | `Result<T>` | 所有接口统一返回结构 |
| 全局异常处理 | `@ControllerAdvice` | 统一转译异常与错误码 |
| 参数校验 | JSR 303 | DTO 入参校验 |
| AOP 日志切面 | `@Aspect` | 记录核心接口入参、耗时、结果 |
| 异步任务 | `@Async` | 承载通知、审核、推荐刷新等任务 |
| 事务管理 | `@Transactional` | 确保核心写操作一致性 |

统一响应建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

错误码分层建议：

- `1000-1999`：参数与请求错误
- `2000-2999`：认证授权错误
- `3000-3999`：业务状态错误
- `5000-5999`：系统内部错误

## 8. 与当前项目基线的差异

当前仓库只包含基础的 Web、JDBC、Redis 依赖。按本文档推进实现时，建议补充以下能力：

1. 引入 MyBatis 及其分页/映射配置。
2. 引入 Spring Security、JWT 相关依赖。
3. 引入 Bean Validation。
4. 引入 Caffeine 本地缓存。
5. 引入 Actuator、AOP、WebSocket。
6. 增加统一配置文件分环境管理。

## 9. 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 架构形态 | 单体架构 | 适合当前项目起步阶段，部署和联调成本低 |
| 认证方式 | JWT + RBAC | 支持前后端分离，便于权限扩展 |
| 数据访问层 | MyBatis | 便于复杂查询、分页和性能优化 |
| 缓存策略 | Caffeine + Redis + 布隆过滤器 | 同时兼顾访问速度、共享状态和防穿透 |
| 推荐模块实现 | 规则推荐起步，异步演进 | 快速上线，后续可平滑升级 |
| 审核策略 | 自动审核 + 人工复审 | 在效率与合规之间取得平衡 |

## 10. 实施计划

1. 第一阶段：完成基础工程搭建，补齐 Security、MyBatis、Validation、Caffeine、Actuator 等依赖。
2. 第二阶段：优先落地用户中心、小说管理、阅读模块，形成主链路闭环。
3. 第三阶段：补齐社区互动、统一异常、统一响应、日志切面、文件上传。
4. 第四阶段：接入 Redis 缓存与本地缓存，完成热点接口缓存治理。
5. 第五阶段：实现推荐引擎与审核引擎的规则版，接入异步任务。
6. 第六阶段：补充监控、运营后台、限流规则与压力测试。

## 11. 风险与注意事项

1. 章节正文可能成为数据库与缓存的双重热点，必须尽早做缓存分层与大文本访问控制。
2. 评论、点赞、收藏等互动行为容易被刷量，需要限流与审计日志联动。
3. 推荐结果的实时性和复杂度要分阶段推进，避免在初期引入过重算法链路。
4. 审核模块必须保留快照与审计记录，否则无法支撑后续申诉与风控追踪。
5. 文件存储虽然初期可本地化，但接口层必须抽象，避免未来切对象存储时改动过大。
